## The duplex deadlock, isolated but not fixed

Worth 7 tests and the only change here with leverage:

```js
const [c, s] = duplexPair();
const ct = tls.connect({ socket: c, ca, host: 'agent1' });      // created FIRST
const st = new tls.TLSSocket(s, { isServer: true, key, cert });
```

Bun writes a 245-byte ClientHello onto the wire, the server's handler **does receive it**, and the engine then produces nothing ‚Äî the handshake stalls forever. Node completes both directions. Narrowing rules out the obvious causes:

- client-side TLS over a Duplex bridged to a real socket: **works**
- server-side TLS over a Duplex bridged to a real socket: **works**
- both ends on a `duplexPair()`, **server** wrapped first: **works**
- both ends on a `duplexPair()`, **client** wrapped first: **deadlocks**

So it is not a generic duplex-TLS gap, and not lost buffered data (`s.readableLength === 0` at wrap time, and `duplexPair` delivers pre-buffered bytes correctly). It looks like re-entrancy in `upgradeDuplexToTLS` (`src/js/node/net.ts:2257`, the `if (!socket)` branch) when the second engine's write synchronously re-enters the first engine's read path. One candidate fix ‚Äî pausing the stream and resuming on `process.nextTick` ‚Äî was implemented, rebuilt, and did **not** help; it was reverted rather than landed on a guess. A real fix needs the native side.
