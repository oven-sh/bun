Two commits: one vendored test, and a native fix for a data-loss bug on the duplex-TLS path.

Stacked on #32630.

## The fix: duplex bytes arriving before the TLS engine starts were discarded

`js_upgrade_duplex_to_tls` defers `start_tls` to an event-loop task, but the JS caller attaches its `data` listener the instant that function returns. Bytes landing in that window reached `on_internal_receive_data` with `wrapper == None` and were **silently dropped** — the whole method body was gated on the engine already existing.

With both ends of a `duplexPair()` wrapped in-process this deadlocks. Whichever side's StartTLS task runs first writes its ClientHello immediately; the peer's engine is still queued; the ClientHello is discarded and the handshake never advances. Wrapping the server first only *appeared* to work because it won the queue.

```js
const [c, s] = duplexPair();
const ct = tls.connect({ socket: c, ca, host: 'agent1' });       // client first
const st = new tls.TLSSocket(s, { isServer: true, key, cert });  // → deadlock
```

Verified both directions on this branch: without the fix the repro prints `DEADLOCK (no secureConnect in 4s)`; with it, `HANDSHAKE OK`.

The fix stages those bytes and replays them from `DuplexUpgradeContext::run_event` once the StartTLS branch completes, so the replay is ordered exactly like an ordinary post-start delivery. They are fed in bounded slices because `receive_data` casts the length through a panicking `expect` — one concatenated buffer would turn a large pre-start burst into a process abort.

**Isolated before any code was changed**: delaying the ClientHello by a single `setImmediate` made it pass (`delay=0: TIMEOUT`, `delay=1/2/3: OK`), which pinned it to the pre-engine window rather than the re-entrancy in `upgradeDuplexToTLS` that was originally suspected.

**Known gap left in place:** a pre-engine `on_end` is still dropped. That needs its own change and is deliberately not bundled here.

## This converts zero tests, and I want that stated plainly

The 7 tests thought to be blocked on this deadlock each have further independent gaps behind it:

| test | before | after |
|---|---|---|
| destroy-stream, destroy-stream-12 | timeout | fails fast — `tls.createServer` connection callback never fires |
| generic-stream | fail | now runs well past the handshake; fails on `_handle.writeQueueSize > 0` (bun writes synchronously) and write-callback ordering |
| snicallback-without-server | fail | `SNICallback` is simply not wired on the duplex path — confirmed unimplemented on both binaries |
| streamwrap-buffersize, connect-stream-writes, handshake-exception | timeout | unchanged; handshake-exception's callback now throws but the exception is swallowed and the loop stays alive |

None of the 7 were vendored, since they would be red.

## Regressions: none, and two small improvements

A/B over the 260 vendored `test-tls-*` / `test-https-*` files shows exactly 2 diffs, both improvements above.

| suite | base | with fix |
|---|---|---|
| node/tls | 193p 4f | 193p 4f |
| node/net | 192p 1f | 192p 1f |
| node/http | 479p 9f | **480p 8f** |
| fetch | 10714p 37f | **10727p 24f** |
| websocket | 144p 17f | **145p 16f** |

One scare worth recording: an early run showed http2 going 305/0 → 303/2. Interleaved 6×6 sampling on an idle machine settled it — the baseline fails the same two test names (`header names longer than 4096 bytes`, `goaway without debug data`) in 2 of 6 runs. Pre-existing http2 flakes; the first measurement was contaminated by two sweeps running concurrently.
