# Consumer-requirements inventory: "fake socket" transports (UpgradedDuplex / WindowsNamedPipe)

All paths relative to `/root/bun/.claude/worktrees/bridge-cse_01UFHwYwi313BkrKbqkCyJeU/`.

## 1. What these types are

Both are **user-space transports that masquerade as `us_socket_t`** only at the `InternalSocket` enum level — they never touch a real fd/poll/uSockets loop.

- **`UpgradedDuplex`** (`src/runtime/socket/UpgradedDuplex.rs:27-44`): TLS over an arbitrary JS Duplex stream (`node:tls` `tls.connect({socket})`, http2 duplex upgrades). Event source is **JS Duplex events**: `get_js_handlers` (`UpgradedDuplex.rs:272-319`) returns a 4-element JS array of host functions `[onData, onEnd, onWritable, onClose]` (`:548-637`) that JS (net.ts) wires to the duplex's `data/end/drain/close` events. Outbound bytes call the duplex's JS `write`/`end` methods (`call_write_or_end`, `:164-201`). No fd, no poll, no `us_loop_t`.
- **`WindowsNamedPipe`** (`src/runtime/socket/WindowsNamedPipe.rs:57-88`): socket-like wrapper over a **libuv `uv_pipe_t`** (module doc `:1-19`). Read side = `uv_read_start` via `uv::StreamReader` trait impl (`:1447-1474`); write side = `bun_io::StreamingWriter<WindowsNamedPipe>` (`:75`, parent vtable at `:1428-1441`).
- **Vtable emulation**: not a C vtable — dispatch is the Rust enum match. `InternalSocket` (`src/uws_sys/socket.rs:56-65`) has `UpgradedDuplex(*mut UpgradedDuplex)` and `Pipe(*mut WindowsNamedPipe)` variants (Pipe payload-less on non-Windows). The `on_socket!` macro (`socket.rs:148-184`) is the de-facto vtable: every `NewSocketHandler` method matches five arms. Because `bun_uws_sys` sits below `bun_runtime`, the payload types are **opaque ZST handles with link-time-dispatch shims**: `src/uws_sys/lib.rs:172-330` declares `extern "C"` symbols `UpgradedDuplex__{ssl_error,is_established,is_closed,is_shutdown,ssl,set_timeout,flush,encode_and_write,raw_write,shutdown,shutdown_read,close}` and the same set + `pause_stream/resume_stream` for `WindowsNamedPipe__*`; the real bodies are emitted by `#[uws_callback(export=…)]` in the runtime tier (e.g. `UpgradedDuplex.rs:216,382,391,397,404,411,418,426,434,446,492`; `WindowsNamedPipe.rs:523,546,562,1107,1140,1146,1195,1241,1255,1264,1272,1284,1328`), plus hand-written `__ssl` shims (`UpgradedDuplex.rs:653-661`, `WindowsNamedPipe.rs:1415-1423`).
- **TLS engine**: both embed `SSLWrapper<*mut Self>` from `bun_uws::ssl_wrapper` (`src/uws/lib.rs:150-1268`), re-exported at `src/runtime/socket/mod.rs:38-39`. `UpgradedDuplex.rs:57`, `WindowsNamedPipe.rs:53`.

## 2. us_* / Loop / SocketGroup interactions; is_named_pipe()/is_detached() branches

**They have almost none of the real uSockets surface:**
- No `us_socket_t`, no `SocketGroup`, no `us_loop_t`. `NewSocketHandler::ext()` and `group()` return `None` for these variants (`uws_sys/socket.rs:612-633`); `fd()` returns `Fd::INVALID` (`:637-642`); `local/remote_port/address` → 0/None (`:644-682`); `set_no_delay/set_keep_alive/set_tos/get_tos` → false/-EBADF (`:517-550`); `sni_resolve` frees the ctx instead (`:555-566`); `start_tls`/`mark_needs_more_for_sendfile` are Connected-only (`:571-575, 684-689`).
- Shared C symbols they *do* use: `us_bun_verify_error_t` struct (`UpgradedDuplex.rs:447-463`), and via SSLWrapper: `us_ssl_socket_verify_error_from_ssl`, `us_get_shared_default_ca_store`, `us_ssl_enable_pending_events`, `us_ssl_pop_pending_session/keylog` (`uws/lib.rs:1239-1267`), `us_ssl_ctx_from_options` via `BunSocketContextOptions::create_ssl_context` (`uws/lib.rs:510-531`).
- Loop: they use the **runtime event loop**, not `us_loop_t` — `EventLoopTimer` on the VM timer wheel via `timer_all()` (`UpgradedDuplex.rs:465-490`, `WindowsNamedPipe.rs:1302-1332`), timer tags `EventLoopTimerTag::{UpgradedDuplex,WindowsNamedPipe}` (`src/event_loop/EventLoopTimer.rs:188,190`) fired from the dispatch table `src/runtime/dispatch.rs:1004-1025`. WindowsNamedPipe additionally uses `vm.uv_loop()` (`WindowsNamedPipe.rs:762,828,872,1097-1105`) and `vm.uws_loop()` only through the StreamingWriter vtable macro (`:1437`).

**Dispatch arms in `uws_sys/socket.rs` handling duplex/pipe:** `write_check_error :263`, `is_closed :272`, `is_shutdown :282`, `is_established :292`, `get_verify_error :306`, `get_error :315` (returns `ssl_error().error_no`), `close :338`, `shutdown :348`, `shutdown_read :358`, `write :370` (→ `encode_and_write`), `write_fd :386-396` (fd **silently dropped**, falls back to plain write), `raw_writev :401-428` (sequential raw writes), `raw_write :431`, `flush :440`, `timeout :452`, `set_timeout :463-483` (no long-timeout wheel — duplex/pipe take plain seconds), `set_timeout_minutes :485` (minutes*60), `pause_stream/resume_stream :497-515` (duplex → `false`, `// TODO: pause/resume upgraded duplex` at `:502,512`; pipe → uv read_stop/read_start), `get_native_handle :590-607` (`ssl()` cast when `IS_SSL`, else `None`).

**`is_named_pipe()` branch sites** (def `uws_sys/socket.rs:103-108, 255-257`): `socket_body.rs:4646,4648` (`js_is_named_pipe_socket` host fn exposed to JS). It also feeds `needs_deref = !is_detached()` semantics indirectly. **`is_detached()`**: pervasive guard in the JS socket layer — ~40 sites in `socket_body.rs` (682, 706, 835, 889, 922, 948, 1056 `needs_deref = !detached`, 1356, 1499, 1572, 1626, 1830, 1845, 1892, 2045, 2110, 2127, 2154, 2197, 2234, 2250, 2267, 2301, 2310, 2327, 2359, 2446, 2474, 2976, 3089-3129, 3211, 4169, 4262), `tls_socket_functions.rs:873,919,1146,1274`, `http/lib.rs:1179`, `Listener.rs:1428`, `Channel.rs:389,517`, `WebSocketUpgradeClient.rs:186`. `detach_for_reconnect` (`socket_body.rs:1256-1290`) explicitly notes Pipe/UpgradedDuplex back-pointers do NOT live in the ext slot (`:1265-1267`).

**Embedding & constructors**: `InternalSocket::UpgradedDuplex/Pipe` built via `NewSocketHandler::from_duplex` (`uws_sys/socket.rs:710-714`) and `from_named_pipe` (`:717-721`). Actual callers:

## 3. TLS over a JS stream (UpgradedDuplex + SSLWrapper)

`SSLWrapper` (`uws/lib.rs:212-1228`) is a memory-BIO TLS pump mimicking uSockets' openssl.c (comment `:177-196`):
- **Init**: `init_with_ctx` (`:380-503`) — `SSL_new`, renegotiation policy (client explicit / server never, `:403-457`), client force-verify with `always_continue_verify` + shared default CA store when CTX is VERIFY_NONE (`:430-444`), two mem BIOs with EOF=-1 (`:460-479`), opt-in session/keylog parking (`:487-490`). `init_from_options` builds CTX from `BunSocketContextOptions` (`:510-531`).
- **Handshake**: `start()` fires `on_open`, then `handle_traffic` (`:533-538`); `update_handshake_state` (`:853-948`) drives `SSL_do_handshake`, sets `authorized`, fires `on_handshake(success, us_bun_verify_error_t)`; verify error via `us_ssl_socket_verify_error_from_ssl` (`:841-850`). Renegotiation rate-limit (3 per 600s, `:208-210, 997-1030`).
- **Data path**: `receive_data` = BIO_write encrypted → `handle_traffic` (`:721-740`); `write_data` = SSL_write plaintext, WANT_READ/WANT_WRITE → pump; fatal → `trigger_close_callback` (`:743-790`); `handle_traffic` loop = handshake → `handle_writing` (BIO_read → `handlers.write` = ciphertext out) ⇄ `handle_reading` (SSL_read → `handlers.on_data` plaintext) (`:1144-1170`), then `flush_pending_events` for session/keylog (`:1178-1219`).
- **Shutdown**: 2-step close_notify vs `fast_shutdown` full-teardown semantics (`:562-665`); `shutdown_read` fakes read-shutdown by swapping `on_data` for a dummy (`:548-554`). `deinit()` idempotent; also runs from `Drop` (`:795-805, 1222-1228`).
- **UpgradedDuplex wiring**: `start_tls` (SSLConfig path, `UpgradedDuplex.rs:321-343`) / `start_tls_with_ctx` (adopted memoised SecureContext `SSL_CTX*` with errdefer free, `:349-380`). Handler fns `on_open/on_data/on_session/on_keylog/on_handshake/on_close` and `write=internal_write` take `*mut Self` (ctx = self backref, `:326-338`). `internal_write` (`:203-214`) resets the timeout then calls the JS duplex's `write(buffer)`/`end(null)` (`call_write_or_end :164-201`; guards: VM shutdown, origin present, callable). Inbound: JS `onData` host fn requires an ArrayBuffer view else emits Node's `ERR_STREAM_WRAP` "Stream has StringDecoder set or is in objectMode" (`:560-578`), then `on_internal_receive_data` → `reset_timeout` → `wrapper.receive_data` (`:223-231`).
- **Timeout emulation**: no uSockets sweep — a per-object `EventLoopTimer` on the VM heap; `set_timeout_in_milliseconds` removes/reinserts (`:465-490`); `on_timeout` checks CANCELLED/VM-not-running before dispatching (`:233-249`). `NewSocketHandler::set_timeout` maps the >240s long-timeout split to plain ms for these transports (`uws_sys/socket.rs:463-493`).
- **Loop liveness**: a TLS engine over a JS stream deliberately does **not** hold the event loop open (`socket_body.rs:4617-4622`).

## 4. WindowsNamedPipe: uv_pipe integration & lifecycle

- **Pipe ownership dance**: `from()` leaks a `Box<uv::Pipe>` and keeps a non-owning `NonNull` alias (`WindowsNamedPipe.rs:612-638`); `start()` transfers ownership to `writer.source` via `start_with_pipe` and sets `PIPE_ADOPTED` (`:1040-1091`); pre-adoption error paths reclaim via `discard_unadopted_pipe` → `uv::Pipe::close_and_destroy` (`:214-228`), covering both never-init'd (`loop_==null` → direct free) and init'd (uv_close then free-in-callback) states; `Drop` reclaims only if `!PIPE_ADOPTED` (`:1385-1408`).
- **Connect/open/accept**: `connect(path,…)` — `uv_pipe_init` + `uv_pipe_connect` with careful single-provenance raw projections for `connect_req`/ctx (`:854-915`), callback `uv_on_connect` → `on_connect` (`:653-715`); `open(fd,…)` — `uv_pipe_open` on an adopted fd then synthesizes `on_connect(0)` (`:812-852`); `get_accepted_by(server_pipe, ssl_ctx)` — server-accept path, `uv_accept`, optional `SSL_CTX_up_ref`+wrapper (`:717-810`). TLS ctx selection prefers a prebuilt memoised `SecureContext` `SSL_CTX*` over rebuilding from `SSLConfig` (`init_tls_wrapper :947-995`).
- **Read**: `on_read_alloc` grows `incoming` spare capacity for libuv (`:239-243`); `on_read` commits nread, `mem::take`s the buffer (Stacked-Borrows decoupling), feeds SSLWrapper or raw `on_data` (`:251-313`). `resume_stream`/`pause_stream` = `read_start_ctx`/`read_stop` (`:523-560`).
- **Write**: `StreamingWriter`; disconnected → enqueue in `writer.outgoing`, else `writer.write`; pipe `ref_()` while data pending, `unref()` on Drained; EOF → `writer.close()` (`:315-344, 467-510`). `shutdown` on plain pipe = `writer.end()` half-close so peer sees EOF (`:1230-1238`).
- **IPC usage**: this type itself is not the IPC fd-passing path — `write_fd` (SCM_RIGHTS) is POSIX-only and pipes silently drop the fd (`uws_sys/socket.rs:379-396`); "Windows IPC fd passing goes through libuv pipes instead" per that comment. `SocketGroup::from_fd(..., is_ipc)` is the real-socket path only (`uws_sys/socket.rs:725-753`).
- **Close/deinit ordering**: `on_close` nulls the `pipe` alias FIRST (writer already took/scheduled uv_close on the Box — `:440-465`), sets IS_CLOSED once, fires `handlers.on_close`, then `release_resources` (`:1338-1382`): cancel timer, `read_stop`, explicit inline `close_without_reporting` for the TLS-close_notify path where the uv handle is otherwise still open (long comment `:1348-1362`), drop wrapper only if `!WRAPPER_BUSY`.
- **Deferred deinit**: `WindowsNamedPipeContext` is intrusive-refcounted; count→0 runs `schedule_deinit` which enqueues an `AnyTask` and frees **next tick** in `run_event` (`WindowsNamedPipeContext.rs:23-62, 285-296`) — required because libuv forbids freeing handle storage in-callback before `uv_close` completes (`Listener.rs:1639-1648`). Listening side: `WindowsNamedPipeListeningContext` (`Listener.rs:1590-1777`) owns the server `uv_pipe`, per-listener `SSL_CTX`; `close_pipe_and_deinit` frees in the `uv_close` callback (`:1666-1682`).

## 5. Lifecycle hazards (quoted)

- **Neuter-in-place, don't `= None` (UpgradedDuplex)** — `teardown()` `UpgradedDuplex.rs:508-519`: "Assigning `None` to the `Option` runs `Drop` … but then memmoves a fresh `Option::None` value over the slot, whose payload bytes are stack garbage — the in-flight frame's `Self::r(this).ssl` then reads junk and `flush_pending_events` UAFs into BoringSSL. `deinit()` alone leaves `ssl = None` / `closed_notified = true` readable … the `Option` is dropped for real when the parent `DuplexUpgradeContext` frees on the next tick."
- **WRAPPER_BUSY (WindowsNamedPipe)** — flag doc `WindowsNamedPipe.rs:105-112`: "`receive_data` can synchronously fire `ssl_on_close → on_close → release_resources`, and dropping the wrapper out from under its own `&mut self` is UAF." Pattern (capture `was_busy`, only outermost clears + runs deferred `self.wrapper = None`) repeated at `:270-305, 487-507, 563-589, 689-707, 785-803, 1019-1035, 1110-1133, 1160-1229`; deferral point `release_resources :1372-1380`.
- **noalias/PROVEN_CACHED launder** — `close`/`shutdown` (`WindowsNamedPipe.rs:1148-1156, 1197-1201`) and SSLWrapper `shutdown/update_handshake_state/handle_writing/handle_traffic` (`uws/lib.rs:563-573, 854-864, 1098-1111`): `black_box`-laundered `*mut Self` because handler re-entry writes `flags`/`ssl` that LLVM was caching ("mirrors the cork fix at b818e70e1c57"). `LaunderedSelf` unsafe impl at `uws/lib.rs:371-374`.
- **Dangling pipe alias** — `on_close` `WindowsNamedPipe.rs:444-451`: "the alias is about to dangle. Clear it here so later `pause_mut()` callers … observe `None` instead of a freed pointer."
- **Handler `*mut Self`, never `&mut Self`** — `WindowsNamedPipeContext.rs:136-150`: forming `&mut *this` in an `on_*` handler "would … pop the caller's Unique tag off the borrow stack → Stacked Borrows UB"; likewise `schedule_deinit :49-55` runs "while `WindowsNamedPipe::on_close` still holds a live `&mut (*this).named_pipe`". Same discipline in `on_read` (`WindowsNamedPipe.rs:258-267`) and `StreamReader::on_read` (`:1462-1473`).
- **Ownership**:
  - `UpgradedDuplex` is an **inline field** of heap `DuplexUpgradeContext` (`socket_body.rs:4060-4080`), which owns it; the `TLSSocket` holds it only as a raw `InternalSocket::UpgradedDuplex` pointer. Context freed by next-tick `AnyTask` `run_event → deinit → heap::take` (`:4213-4336`); over-deref/UAF notes on `on_error`/`on_close` refcount handoffs (`:4157-4177, 4188-4206`).
  - `WindowsNamedPipe` is an inline field of refcounted `WindowsNamedPipeContext` (`WindowsNamedPipeContext.rs:25-43`); `create()` takes +1 on the wrapped TCP/TLS socket (`:396-397`), released in `on_close` (`:275-283`) or `Drop` (`:469-480`); `FailAndRelease` errdefer guard for connect/open failures (`:153-171, 408-466`). `open`/`connect` return a **BACKREF interior pointer** `&mut ctx.named_pipe` (`:426-427, 464-465`) — pointer stability of the containing heap allocation is load-bearing.
  - Both context structs are **self-referential** (task.ctx and handlers.ctx point at the containing allocation) and are constructed by leak-then-field-write at a stable address (`socket_body.rs:4506-4600`, `WindowsNamedPipeContext.rs:322-394`).
- **socket_body.rs treats them differently from real sockets**: no ext-slot backpointer (real sockets stash `*mut Owner` in `us_socket ext`; duplex/pipe owners hold the `InternalSocket` pointer instead — `socket_body.rs:1265-1267`); `needs_deref = !is_detached()` refcount contract relies on `InternalSocket::UpgradedDuplex` being set before `start_tls` is queued (`:4165-4176, 4260-4266`); `on_socket!` `else` arms give them 0/no-op for fd/port/address/sockopt surface.

## 6. Migration notes — what a Rust uSockets-core rewrite must provide

1. **A virtual-transport seam at the handle level, not inside the core.** Today's design already keeps both types entirely **outside** the uSockets core: the only integration point is the `InternalSocket` tagged pointer + `on_socket!` dispatch (`uws_sys/socket.rs`). The cheapest migration is to preserve exactly that: a new core needs only (a) a stable `*mut us_socket_t`-equivalent for the Connected/Connecting arms, and (b) tolerance for enum arms it never sees. Alternatively a `Socket` trait/vtable — but note the handle must stay `Copy` + word-sized (doc `uws_sys/socket.rs:49-54`), and the current layered-crate cycle break (opaque ZST + `extern "C"` `UpgradedDuplex__*`/`WindowsNamedPipe__*` symbols, `uws_sys/lib.rs:172-330`) exists only because the payload types live two tiers up; a same-crate trait object would delete ~160 lines of shims.
2. **Method surface the fake transports must be able to implement** (the full `on_socket!` row set): `write/raw_write/raw_writev/write_check_error, flush, close/shutdown/shutdown_read, is_closed/is_shutdown/is_established, get_verify_error/get_error (ssl_error), set_timeout (seconds; no long-timeout wheel), pause/resume, get_native_handle→SSL*`. Everything fd-flavored (`ext`, `group`, `fd`, ports, addresses, nodelay/keepalive/tos, sendfile, `write_fd`, `sni_resolve`) must be permitted to return None/0/-EBADF.
3. **Keep `SSLWrapper` shared and core-independent.** It lives in `bun_uws` but its only C dependencies are BoringSSL plus five small uSockets helpers: `us_ssl_socket_verify_error_from_ssl`, `us_get_shared_default_ca_store`, `us_ssl_enable_pending_events`, `us_ssl_pop_pending_session/keylog`, and `us_ssl_ctx_from_options`/`BunSocketContextOptions` + `us_bun_verify_error_t` (`uws/lib.rs:1239-1267, 510-531`). The rewrite must re-export equivalents (or move them into the SSLWrapper module) so the fake transports keep TLS without the core. Also `us_get_default_ciphers` (`uws/lib.rs:129-145`).
4. **Timeouts stay on the runtime timer wheel.** Neither type needs `us_timer_t`; they need `EventLoopTimer` tags + dispatch (`dispatch.rs:1004-1025`). The core rewrite need not provide timeout service for virtual transports — but `NewSocketHandler::set_timeout`'s 240s split must remain Connected-only.
5. **Re-entrancy contract**: any handler-vtable callback out of the core (or out of SSLWrapper) may synchronously re-enter write/flush/close/shutdown and may tear down the caller. The rewrite must either keep the WRAPPER_BUSY/launder patterns (`WindowsNamedPipe.rs:105-112`, `uws/lib.rs:563-573`) or — better — design the new API so close is always deferred (queued) rather than synchronous, which would let both files delete their deferred-drop machinery.
6. **Pointer stability**: `InternalSocket::UpgradedDuplex/Pipe` store raw interior pointers into heap contexts (`WindowsNamedPipeContext::open/connect` return `addr_of_mut!((*this).named_pipe)`; `DuplexUpgradeContext.upgrade`). Any new API taking ownership of transports must not move them; deferred (next-tick) deinit of the containing allocation must remain possible (`schedule_deinit`, `deinit_in_next_tick`).
7. **`us_bun_verify_error_t` ABI**: exactly `{int error; const char* code; const char* reason}` (`UpgradedDuplex.rs:460-462`); the fake transports own the string storage (`CertError` `Box<CStr>`), so the type (or its replacement) must permit caller-owned static-lifetime-ish strings.
8. **Windows-specific**: the pipe transport needs continued access to the process libuv loop (`vm.uv_loop()`), `StreamingWriter` parent vtable hooks incl. `uws_loop`/`uv_loop`/`ref_`/`deref` (`WindowsNamedPipe.rs:1428-1441`), and server-side accept from `WindowsNamedPipeListeningContext` (`Listener.rs:1611-1649`) — the `Listener` enum has a `NamedPipe(NonNull<…>)` arm (`Listener.rs:101-107`) that the new listen API must keep representable.
