### 4. EXP-051 remediation: triangulation says compatibility-first checked conversion

The Phase 8 v2 triangulation (`phase8_triangulation.md`) ran external Codex,
external Gemini, and Claude fallback on EXP-051. It found that the v1
"coexistence" recommendation was still a source break: changing a safe method
to `unsafe fn` requires every plugin author to edit call sites at the next
`cargo update`, and it still does not remove UB for callers who blindly wrap
the old method in `unsafe`.

| Lens | Choice |
|------|--------|
| Codex (external) | **D** ŌĆö keep legacy signature, remove transmute via checked path, add `try_output_loader` |
| Gemini (external, flash fallback) | B ŌĆö mark legacy method `unsafe fn` + add `try_output_loader`, but its dissent concedes this does not eliminate misuse |
| Claude review | **D** ŌĆö same as Codex; safe-to-unsafe is a source break |

**Recommended remediation:** the fix *mechanism* (`#[repr(u8)] + try_from`,
hand-written or bytemuck-derived) is acceptable, but the public API must use
option D, not a flag-day return-type swap and not safe-fn-to-unsafe-fn.

1. Keep `pub fn output_loader(&self) -> BunLoader` source-compatible and safe.
2. Replace its body with a checked conversion. Invalid host bytes must take a
   defined path: precise panic or sentinel default + warning. Both are sound;
   the choice is policy.
3. Add `pub fn try_output_loader(&self) -> Result<BunLoader, InvalidLoader>`
   as the new recommended API.
4. Deprecate the legacy method with migration guidance and plan removal or
   tightening in the next major if maintainers want a fallible-only surface.

The C++-mirror question is moot ŌĆö the C++ side already treats `loader` as `u8`, so `#[repr(u8)]` on the Rust side *improves* ABI fidelity. **`bun-native-plugin-rs` is published on crates.io**; flag-day public-FFI breaks and safe-to-unsafe method changes are ecosystem events.

## Methodology Lineage

The prior [`/rust-unsafe-code-exorcist` audit](../../.unsafe-audit/EXECUTIVE_SUMMARY.md) applied 5 passes against Bun and produced 6 ceiling-score supply-chain attack primitives + 40 T1/T1-equivalent findings + 5 Miri-backed UB witnesses, distilled from 11,044 unsafe sites across 108 workspace crates. It opened two PRs ([#30763](https://github.com/oven-sh/bun/pull/30763) audit artifacts, [#30765](https://github.com/oven-sh/bun/pull/30765) three highest-confidence fixes).

This UB audit treats those findings as Phase-0 anchored witnesses (`phase0_run.json:known_ub_witnesses_to_anchor`) and re-verifies them under the pinned nightly + audited source. It then runs the broader UB taxonomy in `references/UB-TAXONOMY.md` (25 buckets) against the entire Rust workspace, surfacing new finding classes the unsafe-code pass did not target:

- Bucket-14 constŌåÆmut count writes via `addr_of!.cast_mut()` on `&self` (12 sites)
- Bucket-7 Cell-cross-thread races under mutex serialisation (web_worker, WatcherAtomics, ThreadPool::Queue, UnboundedQueue)
- Bucket-11 panic-safety leaks across FFI boundaries (`AnyTaskJob`, Listener `ptr::read+forget` window)
- Bucket-10 FFI layout-assert drift in NAPI / Windows / BoringSSL mirrors
- Bucket-2 strict-provenance Cluster D (true TaggedPtr helper + custom packed-pointer sites + reviewed FFI/layout-only rows)
- Bucket-15 lifetime-escape cluster (17-site `fn(&self) -> &'a mut T` + `source_writer_escape() ŌåÆ &'static mut Writer`)
- Bucket-5 uninit-slot family extension (HiveArrayFallback 8-site caller migration)

The unsafe-code audit asked "is this `unsafe` block necessary?"; this audit asks "is the behaviour at this site actually defined under Rust's memory model?" ŌĆö orthogonal questions, complementary outputs.
