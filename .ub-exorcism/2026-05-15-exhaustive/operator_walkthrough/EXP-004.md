# Operator-Library Walkthrough — EXP-004

**Target:** `webcore/encoding.rs` `Vec<u8>→Vec<u16>` allocator-layout mismatch on dealloc.
**Source of truth:**
- Registry entry — `.ub-exorcism/2026-05-15-exhaustive/UNDEFINED_BEHAVIOR_EXPERIMENT_DESIGNS.md`, lines 189-221.
- Remediation rubric — `.ub-exorcism/2026-05-15-exhaustive/phase8_remediation_plan.md`, lines 208-223.

This walkthrough reconstructs the live hunt for EXP-004 as a sequence of five
ritual operators (★ SUSPECT, ♦ COUNTER, ⊳ READ-ONLY-DELTA, ☣ SAFETY-NOTES-FIRST,
⊳ FIX-PREP). Each operator's text is exactly what the prompt to the next pass
would look like if this were happening live, so a reader can mechanically apply
the same flow to a new candidate without re-deriving the structure.

---

## 1. OPENING — ★ SUSPECT (numbered category audit)

★ SUSPECT operators are **bucketed**: they name a UB *category* rather than a
specific failure. They cast a wide net so the next layer (♦ COUNTER) can prune.

> **★ SUSPECT-20 (Dangling Box / allocator pairing).** Audit
> `src/runtime/webcore/encoding.rs` end-to-end for any site that constructs an
> owned `Vec<T>` / `Box<T>` from raw parts where the element type used for
> deallocation differs from the element type used at allocation. The
> `GlobalAlloc` contract requires the dealloc `Layout` to match the alloc
> `Layout`; an element-type swap silently changes either `size` or `align` and
> drops UB into the runtime even when mimalloc happens to absorb it. Enumerate
> every `Vec::from_raw_parts`, `Box::from_raw`, `slice::from_raw_parts_mut`
> followed by `Vec::from_raw_parts`, and any `ManuallyDrop`-wrapped owner that
> hands ownership to a different typed container.

This is exactly **Bucket 20** of the project bucket list (cross-referenced in
phase8_remediation_plan.md line 1252 — "Bucket-3/20 typed-slice / Vec-layout
reinterpretation"). The operator does **not** yet name a specific function,
line, or transmute — that pruning is the COUNTER's job.

## 2. PIVOT — ♦ COUNTER (named failure mode)

The ★ SUSPECT-20 sweep over `encoding.rs` returns three call sites that touch
`from_raw_parts` (the `Hex` arm, the `Latin1` arm via `create_uninitialized_latin1`,
and the `Ucs2 | Utf16le` arm). Most of them deallocate through the same element
type they allocated with. The COUNTER picks the *one specific mode of failure*
worth burning a Miri witness on.

> **♦ COUNTER (Vec-rep reuse across element-width).** Could the `Ucs2 |
> Utf16le` arm be reusing a `Vec<u8>` allocation as a `Vec<u16>` allocation?
> Specifically: does it call `Vec::from_raw_parts::<u16>` over the byte
> buffer's pointer with `capacity / size_of::<u16>()`, leave the original
> `Vec<u8>` un-dropped, and then let the resulting `Vec<u16>::drop` call
> `dealloc(size=N, align=2)` over a region that was allocated with
> `dealloc(size=N, align=1)`?

Note the shape of this question. It is **not** "is this safe?" — that
question elicits "the SAFETY comment looks fine." It names **two parties**
(the byte allocation, the wide-element drop) and **one specific race / mismatch**
(allocator-layout pairing on dealloc), which makes the answer falsifiable:
either the layouts match or they don't, and Miri's
`-Zmiri-symbolic-alignment-check` will decide.

## 3. CONFIRM — ⊳ READ-ONLY-DELTA (do NOT edit any files)

Before drafting a fix, the operator forces a freeze: re-read the actual source
and confirm the shape verbatim. No mutations.

> **⊳ READ-ONLY-DELTA.** `git diff --stat src/runtime/webcore/encoding.rs` —
> expect zero changes from base `4d443e5402`. Then `git show
> 4d443e5402:src/runtime/webcore/encoding.rs | sed -n '295,312p'` and paste
> the raw lines. **Do NOT edit any files at this step.**

Result (verbatim from current tree, `encoding.rs:298-310`):

```rust
            // TODO(port): Zig reinterpreted the owned u8 allocation as []u16 (with @alignCast)
            // and handed it to createExternalGloballyAllocated(.utf16, ...). Reinterpreting a
            // Vec<u8> as Vec<u16> is not generally sound in Rust (alignment + allocator layout).
            // Phase B: route through bun_core::String API that accepts raw (ptr,len,cap) bytes.
            // SAFETY: input.as_ptr() is at least 1-aligned; Zig asserted u16 alignment via @alignCast.
            let as_u16 = unsafe {
                let mut input = core::mem::ManuallyDrop::new(input);
                Vec::from_raw_parts(
                    input.as_mut_ptr().cast::<u16>(),
                    usable_len / 2,
                    input.capacity() / 2,
                )
            };
            create_external_globally_allocated_utf16(as_u16)
```

The shape matches the COUNTER hypothesis exactly: `ManuallyDrop` suppresses the
`Vec<u8>` drop (no double-free), the cast is `u8 → u16`, and the resulting
`Vec<u16>` will eventually be dropped (handed to
`create_external_globally_allocated_utf16` which transfers ownership to
WebKit's external buffer, whose finalizer ultimately calls `Vec<u16>::drop`).

**Falsifier check from registry (line 212):** "mimalloc is more permissive in
practice than the abstract `GlobalAlloc` contract; production binaries may not
crash. But Miri's abstract-machine verdict is the authority here." → The
in-source `TODO(port)` block (lines 298-301) explicitly names alignment +
allocator-layout as the soundness gap, **author admits the cast is unsound**
(registry line 220 — PRESENT_WEAK SAFETY comment). Pivot to Miri to harden.

Witness recorded: registry line 219 — Miri reports
`incorrect layout on deallocation: allocN has size 6 and alignment 1, but
gave size 6 and alignment 2`. Verdict: **CONFIRMED_UB**.

## 4. INVARIANT-CAPTURE — ☣ SAFETY-NOTES-FIRST (before any code)

Before letting any code be generated, write the invariants the fix must
preserve. This blocks "drive-by" rewrites that pass Miri but break ABI,
ownership, or behaviour.

> **☣ SAFETY-NOTES-FIRST for R-EXP-004.** Any fix MUST preserve, document, or
> explicitly relax (with rationale) each of the following invariants:
>
> 1. **Allocator-layout pairing.** Any `Vec<U>` (or owning container) handed
>    downstream for eventual `drop` MUST have been allocated with a `Layout`
>    matching what `Vec<U>::drop` will request — i.e. `size_of::<U>() * cap`
>    and `align_of::<U>()`. A `u8` allocation cannot be dealloc'd as `u16`.
> 2. **Odd-length input.** Current code clamps `usable_len = input.len() &
>    !1` (lines 287-291). The fix MUST still tolerate odd-length inputs and
>    drop the trailing byte; this is observable behaviour for the `Ucs2 |
>    Utf16le` JS-visible decoder.
> 3. **Empty input.** Empty input MUST return `BunString::empty()` (current
>    line 295), not allocate.
> 4. **Endianness.** The decoder is `Utf16le` / `Ucs2`. On little-endian
>    targets `u16::from_ne_bytes` and `u16::from_le_bytes` agree; on
>    big-endian targets the fix MUST still produce little-endian-decoded code
>    units. Either pick `u16::from_le_bytes` explicitly or assert
>    `cfg(target_endian = "little")`.
> 5. **Ownership transfer to WTF.** `create_external_globally_allocated_utf16`
>    consumes the `Vec<u16>` and hands it to a JSC external buffer whose
>    finalizer eventually frees it. The fix MUST hand a `Vec<u16>` that was
>    allocated by Rust's global allocator (`#[global_allocator] = mimalloc`)
>    so the finalizer reaches the same allocator. Do not switch to a
>    `Box<[u16]>` or a `bun_core::String::clone_utf8` path without
>    re-validating that callsite.
> 6. **No double-free.** The original `Vec<u8>` is currently neutered via
>    `ManuallyDrop`. The fix MUST either drop the `Vec<u8>` exactly once (if
>    we allocate a fresh `Vec<u16>` and copy bytes) or keep a single owner —
>    not both.
> 7. **Cross-thread string hazard (CLAUDE.md §JSC Interop).** The result is
>    not an `AtomString`; it goes through `createExternalGloballyAllocated`
>    which produces a plain `WTFStringImpl`. The fix MUST NOT introduce an
>    `AtomString` path here (the decoder runs on the JS thread, but the
>    resulting `BunString` may flow to non-JS threads via
>    `Response::init`-style code paths — see
>    `src/runtime/webcore/fetch/FetchTasklet.rs`).

These invariants are derived from registry notes (lines 219-221), CLAUDE.md
§Strings, CLAUDE.md §Cross-thread string hazards, and direct inspection of
`encoding.rs:283-312`. Writing them down **first** means the fix prompt can
be evaluated against a checklist rather than "looks fine."

## 5. STAGE — ⊳ FIX-PREP (read-only delta for the fix design)

Same discipline as ⊳ READ-ONLY-DELTA, but applied to the *fix surface* so the
fix prompt knows the blast radius before writing a single line.

> **⊳ FIX-PREP.** Do not edit yet. Enumerate (a) files the fix would touch,
> (b) tests that must pass, (c) the candidate winner from phase8 with
> rationale, (d) the witness that proves the new soundness.

**(a) Files the fix would touch:**

  block. Single function-local change; the rest of the function is
  untouched.
- No other files. `create_external_globally_allocated_utf16` already accepts
  a `Vec<u16>` by value and is untouched.

**(b) Tests that must pass:**

  especially `utf-16le` / `ucs-2` decode paths.

- `test/regression/issue/<no-number>` — none required; this is not a
  user-visible regression, it's a Miri-only UB.
- Odd-length input regression: a UTF-16LE decode of a 5-byte buffer must
  produce 2 code units and silently drop the trailing byte.
- Empty-input regression: `new TextDecoder('utf-16le').decode(new
  Uint8Array(0))` must return `""`.
- Allocator witness: `experiments/EXP-004/src/main.rs` rewritten to mirror
  the *fixed* shape; expect zero Miri layout complaint.

**(c) Candidate winner from phase8 rubric (lines 213-219):**

| ID | Description | Total |
|----|-------------|------:|
| A | Allocate fresh `Vec<u16>` and copy via `chunks_exact(2).map(u16::from_ne_bytes)` | **18** |
| B | `bytemuck::cast_vec` | 20 if supported |
| C | Keep `from_raw_parts` + manual `Layout::array::<u16>` dealloc | 11 |

**Winner verification (phase8 line 219):** bytemuck does **not** support
`cast_vec` between element sizes (alignment + capacity invariants prevent it
— `cast_slice` + collect collapses to A). **Practical winner = A: explicit
copy via `chunks_exact(2).map(u16::from_le_bytes).collect()`**, with
`from_le_bytes` selected over `from_ne_bytes` to honour Invariant 4
(endianness) on big-endian targets without a `cfg` gate.

**(d) Witness for new soundness (phase8 line 223):**
- New `experiments/EXP-004-fix/src/main.rs` — same shape as
  `experiments/EXP-004/src/main.rs` but using `chunks_exact(2)
  .map(u16::from_le_bytes).collect::<Vec<u16>>()` instead of the
  cross-element `from_raw_parts`. Run under `cargo +nightly miri run` with
  `MIRIFLAGS=-Zmiri-symbolic-alignment-check`; expected output is
  *empty* (no UB diagnostic), i.e. the same shape that EXP-004 *fails* must
  now *pass*. This is the canonical "proves-new-soundness" pattern from
  phase8.

**Blast radius:** strictly local to one function arm in `encoding.rs`. No ABI
change, no public API change, no cross-crate impact. The fix is a 13-line
replacement of one `unsafe { ... }` block with a safe `.chunks_exact(2)
.map(u16::from_le_bytes).collect()` expression.

## Why this flow

The five operators form a funnel:

1. **★ SUSPECT** opens a *category* (Bucket 20) and lists candidate sites.
2. **♦ COUNTER** picks one site and names the *failure mode* concretely.
3. **⊳ READ-ONLY-DELTA** freezes the source state and confirms the COUNTER
   shape is real — no fix-thinking allowed yet.
4. **☣ SAFETY-NOTES-FIRST** captures invariants *before* any code generation,
   so the fix can be evaluated against a checklist.
5. **⊳ FIX-PREP** stages the fix (files, tests, candidate, witness) without
   actually editing.

Only after FIX-PREP does the operator advance to the edit / Miri-rerun /
review loop. Front-loading the constraints is what keeps the "fix" from
silently introducing a new bug class (e.g., a `Box::leak` of a
`Box<[u16]>`, an `AtomString` cross-thread hazard, or a big-endian
behaviour drift) — each of which is invisible without the invariants
written down first.
