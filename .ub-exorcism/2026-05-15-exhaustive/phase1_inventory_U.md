## EXP-013 anchor — status: **STILL APPLIES**

See `phase1_notes/U_crash_meta.md` §EXP-013 for the per-handler call-graph audit. **One-line verdict**: every documented async-signal-safety violation in the prior audit is present in current source, and the in-source TODO at `src/crash_handler/lib.rs:588` (`// TODO: I don't think it's safe to lock/unlock a mutex inside a signal handler.`) confirms the maintainers know about it. The Zig sibling has the same shape; the Rust port carried it forward intact. SA_RESETHAND on the sigaction (line 1737) is the only mitigation. One POSIX signal entry point (`handle_segfault_posix`, line 1657), one Rust panic hook routed through similar reporting code from non-signal contexts, and one Windows entry point (`handle_segfault_windows`, line 2029, in a VEH — a different concurrency model with different rules; not async-signal-safe-bound) feed into the same `crash_handler()` body, which then calls a long list of non-async-signal-safe functions on the POSIX signal path.

## Inline asm — status: **CLEAN, clobber-correct**

3 `core::arch::asm!` sites, all in `src/perf/hw_timer.rs`:
- Line 37: `mrs {ret}, CNTVCT_EL0` — aarch64 virtual-counter read. Clobbers: `ret = out(reg)`. Options: `nomem, nostack, preserves_flags`. Per ARM ARM v8 §C5.2.18, `MRS` to general register has no other architectural effect — clobber list correct.
- Line 51: `rdtsc` — x86_64 timestamp counter read. Clobbers: `out("eax") lo, out("edx") hi`. Options: `nomem, nostack, preserves_flags`. Per Intel SDM Vol 2B §RDTSC, clobbers `eax` + `edx` only (no flags, no other GPRs) — clobber list correct.
- Line 154: `mrs {ret}, CNTFRQ_EL0` — aarch64 counter frequency. Same shape as line 37 — correct.

No `global_asm!` anywhere in U. No SIMD intrinsics with `target_feature` gates beyond `#[cfg(target_arch)]`. **Verdict**: inline asm in U is the textbook minimal correct form.

## Notable patterns

1. **Signal-handler async-signal-safety hazard is well-documented in source.** The crash handler's own line 588 admits the issue; line 1737 sets `SA_RESETHAND` as a partial mitigation. EXP-013 is not a hidden bug — it is a known-and-accepted hazard inherited from the Zig spec. A Phase-2 fix would split the signal entry from the report path: signal handler writes a fixed-size message with `write(2)` only, then re-raises. The current code path through `Output::flush` / `bun_which::which` / `libc::fork` / `libc::execve` is the dangerous fan-out.
2. **`options_types::context.rs` is the safety-comment gold standard.** Three `pub unsafe fn` accessors (`log`, `log_mut`, `log_ref`) carry multi-paragraph `# Safety` docs that name every other aliasing path (transpiler, install, JSON parsing, package manager) and explicitly debunk the "`&mut self` proves exclusivity" intuition. This is the right shape for any future audit of process-global mutable state — Phase-2 review elsewhere should aspire to this.
3. **Inline asm clobber lists are correct everywhere they appear.** All three sites in `perf/hw_timer.rs` use the minimal (`nomem, nostack, preserves_flags`) options + correct out-register declarations. The `core::arch::x86_64::__cpuid_count` intrinsic is used in preference to raw asm for CPUID (PORT NOTE explicitly cites the LLVM `rbx` PIC base reservation as the reason).

## Open questions

1. **EXP-013 priority.** The signal-handler-async-signal-safety hazard is comment-acknowledged but unfixed across both Zig and Rust. Phase-3 should design the experiment shape: can we Miri-witness a deadlock by panicking inside the panic handler under signal delivery? Or does the SA_RESETHAND short-circuit always reach `crash()` first?
2. **Stacked-Borrows status of the Watcher `*mut Self` thread-spawn handoff.** `thread_main` scopes `me = &mut *this` *before* calling `heap::take(this)`. The `# Safety` doc cites the Stacked Borrows protector rule. Should this be a Phase-3 Miri target to confirm the scope is tight enough?
3. **`paths/resolve_path.rs` thread-local `&'static mut`.** `tl_buf_mut` returns `&'static mut [u8; N]` from a thread-local UnsafeCell. Sound because thread-local + UnsafeCell prevents cross-thread aliasing, but the same-thread caller must not hold two such borrows simultaneously. Consider a Phase-2 review for whether all callers obey this.
4. **`options_types/compile_target.rs:78` `from_utf8_unchecked` of `npm_name()`.** The npm name table is ASCII-by-construction but the `# Safety` invariant isn't spelled out at the call site. Phase-2 candidate for an explicit ASCII assertion.
5. **`watcher/INotifyWatcher.rs` Event alignment assumption.** Kernel pads inotify events to 4-byte alignment after the variable-length name; the current code does aligned reads. PORT NOTE at line 104 names the fallback. Worth a Phase-2 sweep on alternative kernels (BPF tracepoint streams have similar shapes).
