# Production-ready bar for this PR

1. The implementation must be fully wired end-to-end: `OverrideMap.get` receiving correct `parent_package_id` at every call site, lockfile serialize/deserialize producing stable output, and no dead code paths remaining.
2. Existing global override/resolution behaviour must be preserved without regression — the same lockfiles produced before must still be produced for global-only overrides.
3. Both the text lockfile (`bun.lock`) and binary lockfile (`bun.lockb`) must round-trip scoped overrides without loss: `save → load → save` must produce an identical lockfile.
4. Tests must cover the real changed paths: scoped override applies under matching parent, scoped override does not apply under different parent, dot+child rules both fire, Yarn-style `resolutions` parse into scoped entries, scoped parent package names work, and lockfile stability is asserted after install.
5. The override-change re-resolution TODO must remain documented and not silently break existing installs — the `parent_package_id = null` fallback must not produce wrong resolutions more often than before this change.
6. Performance must remain acceptable: scoped lookup is an additional hash-map lookup in `OverrideMap.get` only when `parent_package_id` is provided; no extra allocations in the hot resolution path.
7. No secrets, commented-out code, stale comments, or drive-by refactors must be present.

# Findings

## 1. Correctness & functional completeness

### BLOCKING — Override-change re-resolution passes null parent, losing scoped override context

- Type: Verified issue
- Evidence: `src/install/PackageManager/install_with_manager.zig:384-393`
- Why it matters: When `overrides_changed` is true and a lockfile already exists, the flat dependency-buffer scan finds matching child name hashes but cannot determine each dependency's owning parent package. The `enqueueDependencyWithMain` call at line 388 passes `null` for `parent_package_id`, so scoped overrides are not applied during re-resolution. This means editing a scoped override in `package.json` and running `bun install` with an existing lockfile will silently use global resolution instead of the scoped rule.
- Recommendation: Build a temporary `DependencyID → owning PackageID` map before the loop by walking each package's `dependencies` slice from `manager.lockfile.packages.slice().items(.dependencies)`. Then pass `owner[dependency_i]` as `parent_package_id` instead of `null`. This is the minimal fix that makes one-level scoped override re-resolution correct without a persistent lockfile index.
- Confidence: High

### NON-BLOCKING — processPeerDependencyList called with null parent at both call sites

- Type: Verified issue
- Evidence: `src/install/PackageManager/install_with_manager.zig:542`, `src/install/PackageManager/install_with_manager.zig:620`, `src/install/PackageManager/processDependencyList.zig:287-296`
- Why it matters: `processPeerDependencyList` was updated to accept `parent_package_id: ?PackageID` (processDependencyList.zig:287), but both call sites still pass `null`. This means scoped overrides that should apply to peer dependencies based on the package that declares the peer will not work. Whether this is a problem depends on the intended semantics — npm and pnpm do handle peer dependencies in their override application, but this is an intentional scope choice for this initial PR.
- Recommendation: Either (a) thread correct parent context through `processPeerDependencyList` calls if peer overrides should be supported, or (b) document explicitly that peer dependency scoped overrides are not supported in this release and keep the null pass-through.
- Confidence: Medium

### NON-BLOCKING — TaskCallbackList.append passes null for non-dependency items

- Type: Verified issue
- Evidence: `src/install/PackageManager/runTasks.zig:788`, `src/install/PackageManager/runTasks.zig:1043`
- Why it matters: When a callback list entry is not a `.dependency` or `.root_dependency`, the `append` call passes `null` as `parent_package_id` rather than the available `package_id`. This could cause scoped override context loss for any callback type that reaches `processDependencyListItem` without explicit parent plumbing. The current branch handles this correctly for `.dependency` and `.root_dependency` types (runTasks.zig:788 passes `package_id`), but the null fallback for other types is inconsistent.
- Recommendation: Pass `package_id` consistently in `append` calls at both locations (runTasks.zig:788 and runTasks.zig:1043).
- Confidence: Medium

No other correctness issues found in the reviewed diff. The scoped lookup logic in `OverrideMap.get` is correct, the precedence chain is correct, and all other call sites appear properly wired.

## 5. Tests & validation

- Type: Verified issue

- Why it matters: The most significant correctness risk in this PR is the override-change re-resolution path (finding 1 above). None of the tests exercise the scenario where a lockfile exists and a scoped override is then changed. The `ensureLockfileDoesntChangeOnBunI` helper verifies lockfile stability but not override-change behaviour.
- Recommendation: Add a test that: (a) installs with a scoped override, (b) verifies the override applied, (c) changes the scoped override version, (d) reinstalls, and (e) verifies the new version was adopted. This test should fail if the `null` parent re-resolution path is not fixed.
- Confidence: High

## 8. Adversarial review

What assumption is most likely to be wrong? The assumption that every dependency-enqueuing call site now correctly passes `parent_package_id`. The diff confirms this is mostly true for the primary resolution paths, but the override-change re-resolution path, peer dependency processing, and non-dependency callback types all have null parent context. The assumption is partially violated.

What happy-path-only logic is hiding here? The override-change re-resolution loop. It works correctly for fresh installs (when the dependency list is enqueued with proper parent context through `doFlushDependencyQueue`), but fails silently for existing lockfile updates because the flat scan cannot reconstruct ownership.

What would break first in production? A user editing a scoped override in an existing project's `package.json` and running `bun install` would see the old resolution persist, with no error or warning.

What test gives false confidence? `ensureLockfileDoesntChangeOnBunI` — it proves the lockfile is stable across repeated installs, but it does not prove that changing an override triggers correct re-resolution. A reader could mistakenly interpret it as covering override-change behavior.

What part is too implicit and likely to be missed by the next contributor? The fact that `parent_name` in `ScopedOverrideKey` is a serialization-only field that does not participate in hash/equality. This is correct but non-obvious.
