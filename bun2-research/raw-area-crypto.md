# Bun 2.0 candidates - crypto/hashing surface (Bun.hash, Bun.password, Bun.CryptoHasher, Bun.sha/SHA*, Bun.CSRF, Bun.randomUUID*)

All claims below were verified by reading the checked-out source and/or running `bun 1.4.0` directly.

### `Bun.password.hash()`/`verify()` (async) stringify arbitrary input; the sync twins throw
what: The async `Bun.password.hash`/`Bun.password.verify` coerce any non-string, non-buffer argument via `toString()`, while `hashSync`/`verifySync` throw `TypeError`.
where: `/workspace/bun/src/runtime/crypto/PasswordObject.rs:756`, `:859`, `:866` (regret comments); `from_js_to_owned_slice` at `/workspace/bun/src/runtime/node/types.rs:334`; types at `/workspace/bun/packages/bun-types/bun.d.ts:3219-3234`.
evidence: Three verbatim source comments: `// TODO: this most likely should error like \`hashSync\` instead of stringifying.` (PasswordObject.rs:756, 859, 866). Reproduced: `await Bun.password.hash(undefined)` returns a real bcrypt hash of the 9-byte string `"undefined"`, and `Bun.password.verify("undefined", h)` â†’ `true`; `Bun.password.hashSync(undefined)` throws `Expected password to be a string or TypedArray for 'hash'.` Same for `hash(42)` and `hash({})` (â†’ `"[object Object]"`).
why bad: A missing/undefined form field silently becomes the password `"undefined"` - a security bug that the sync variant already prevents. The maintainers have already written down that the async behavior is wrong; they just can't change it without breaking callers who rely on number/object coercion.
bun 2.0 proposal: Make `hash`/`verify` validate arguments identically to `hashSync`/`verifySync` (throw `ERR_INVALID_ARG_TYPE` for non string/TypedArray), and delete the three TODOs.
blast radius: low - only code passing non-string/non-buffer passwords breaks, and that code is almost certainly already buggy.
confidence: high.

### `Bun.CryptoHasher` has two incompatible `digest()` lifecycles in one class, both diverging from `node:crypto`
what: Non-HMAC `Bun.CryptoHasher.digest()` silently resets the instance for reuse; the HMAC variant of the *same class* is consumed and throws on reuse; `node:crypto.Hash.digest()` throws on reuse.
where: `/workspace/bun/src/runtime/crypto/EVP.rs:248` (`self.reset(engine)` inside `final`); `/workspace/bun/src/runtime/crypto/CryptoHasher.rs:760` (`inner.replace(None)`), `:304` (`"HMAC has been consumed and is no longer usable"`); `.d.ts` at `/workspace/bun/packages/bun-types/bun.d.ts:4956` (`"Resets the CryptoHasher so it can be reused."`); `/workspace/bun/docs/runtime/hashing.mdx:298`.
evidence: `docs/runtime/hashing.mdx:298`: "Unlike the non-HMAC `Bun.CryptoHasher`, the HMAC `Bun.CryptoHasher` instance is not reset after `.digest()` is called, and using the same instance again throws an error." Reproduced: non-HMAC `h.digest("hex"); h.update("hello"); h.digest("hex")` returns the same hash twice (silent reset+reuse); `node:crypto` `createHash("sha256")` in the same process throws `Digest already called`; Bun's HMAC variant throws `HMAC has been consumed and is no longer usable`. The split personality exists because #13007 bolted the HMAC key onto the 2nd constructor positional (instead of a separate `CryptoHmac`).
why bad: Code ported from Node that accidentally reuses a `Hash` after `digest()` crashes on Node but silently produces a *different, valid-looking* second hash on Bun. Within Bun itself the same method on the same class has opposite semantics depending on a constructor argument you can't see at the call site.
bun 2.0 proposal: Pick one lifecycle - Node's (throw after `digest()`). If reset-and-reuse is wanted, make it an explicit `reset()` method. Split HMAC into its own class (or `Bun.CryptoHmac`) so the type states the contract.
blast radius: medium - any code relying on the silent-reset reuse breaks loudly (good), but it's easy to miss in a migration.
confidence: high.

### `Bun.sha` + `Bun.SHA1`/`MD4`/`MD5`/`SHA224`/`SHA256`/`SHA384`/`SHA512`/`SHA512_256` are a dead parallel API surface
what: Nine globals (`Bun.sha` plus 8 hash classes) that are fully superseded by `Bun.CryptoHasher`, are no longer documented, and include broken algorithms (MD4, MD5) as named top-level Bun APIs.
where: `/workspace/bun/src/runtime/crypto/crypto.classes.ts:3` (the 8 class names); `/workspace/bun/src/runtime/api/BunObject.rs:213,389-396`; `/workspace/bun/packages/bun-types/bun.d.ts:4854-5172`.
evidence: `Bun.sha("hello","hex") === Bun.CryptoHasher.hash("sha512-256","hello","hex")` (verified identical). The base type's JSDoc admits it is scaffolding: `packages/bun-types/bun.d.ts:4855` - "This class only exists in types". The `SHA1` JSDoc (`bun.d.ts:5102-5104`) reads "This is not the default because it's not cryptographically secure and it's slower than {@link SHA512}. Consider {@link SHA512_256} instead" - a comment about a *default* that no longer exists. `docs/runtime/hashing.mdx` documents only `Bun.password`, `Bun.hash`, `Bun.CryptoHasher`; `docs/runtime/bun-apis.mdx:35` lists a bare `Bun.sha` with no link and omits the classes entirely. The classes predate `CryptoHasher` (added 2022-04-12, commit "Implement Bun.sha1, Bun.sha256â€¦"; `CryptoHasher` added 2022-12-29).
why bad: Nine names on the `Bun` global that do nothing `Bun.CryptoHasher` doesn't, are undocumented, carry stale self-referential doc comments, and expose MD4/MD5 as headline names. `Bun.sha` being hard-wired to SHA-512/256 is an opinionated, non-obvious choice nobody would design today.
bun 2.0 proposal: Delete `Bun.sha`, `Bun.SHA1`, `Bun.SHA224`, `Bun.SHA256`, `Bun.SHA384`, `Bun.SHA512`, `Bun.SHA512_256`, `Bun.MD4`, `Bun.MD5`, and the `CryptoHashInterface` type. Point users at `Bun.CryptoHasher.hash()`.
blast radius: low - they are undocumented and `Bun.CryptoHasher` is a drop-in replacement.
confidence: high.

### `Bun.hash` return type is `number | bigint` depending on which variant, and has already been breaking-changed once for exactly this
what: `Bun.hash()` and its 64-bit sub-algorithms return `bigint`; the 32-bit ones return `number`; the top-level declared type is the union `number | bigint`. `Bun.hash.xxHash3` still silently truncates its seed to 32 bits - the same bug already breaking-fixed twice for its siblings.
where: `/workspace/bun/packages/bun-types/bun.d.ts:2305-2324`; `/workspace/bun/src/runtime/api/HashObject.rs:18-30` (u32â†’number, u64â†’BigInt), `:109-121` (xxHash3 seed truncation); `/workspace/bun/docs/runtime/hashing.mdx:170`.
evidence: PR #4054 ("Fix `Bun.hash` functions", merged 2023-08-08) is an explicit on-record admission that the original design was wrong: "The 64-bit hashes cannot be represented by plain JS numbers and were being truncated, yielding incorrect hashes, they now return bigints. â€¦ the seeds were being truncated to 32 bits â€¦ All hashes were returning signed values while hashes are not meant to be negative." Issue #21879 / PR #21881 fixed the identical 32-bit-seed truncation in `xxHash64`. The same bug survives today in `xxHash3`: `Bun.hash.xxHash3("data", 0n) === Bun.hash.xxHash3("data", 1n<<32n)` (verified on 1.4.0), and the source comment at `HashObject.rs:117-118` reads "The seed is truncated to u32 before widening back to XxHash3's native u64 - preserve that truncation for output stability." A commit on a non-HEAD branch (`9a5d6386dd`, "hash: use the full 64-bit seed in Bun.hash.xxHash3") shows the team already wrote the fix and is holding it because it changes output.
why bad: Callers can't write generic code over `Bun.hash.*`. `typeof Bun.hash("x") === "bigint"` but `typeof Bun.hash.crc32("x") === "number"`. And there is a known-wrong algorithm (`xxHash3` â‰  `XXH3_64bits_withSeed` for any seed â‰Ą 2^32) sitting behind a format-stability guarantee - the canonical case for bundling into a major.
bun 2.0 proposal: Land the xxHash3 seed fix; make every `Bun.hash.*` return `bigint` (or accept an `{asBigInt}`/encoding option) so the return type is uniform.
blast radius: medium - any persisted xxHash3 output or code doing `Number`/`bigint` arithmetic on 32-bit variants changes.
confidence: high.

### `Bun.hash.*` "seed" argument has three different meanings, two of which are silently ignored
what: The second argument to the `Bun.hash.*` family is documented as one uniform "seed" but is (a) a real seed for xxHash*/wyhash/murmur/cityHash64, (b) a chainable running checksum for `crc32`, (c) completely ignored for `adler32` and `cityHash32`. Non-numeric seeds are silently treated as 0. File-backed `Blob`s silently hash as empty.
where: `/workspace/bun/src/runtime/api/HashObject.rs:56-58` ("Single-arg, seed ignored."), `:74-77` ("Single-arg, seed ignored (the JS seed is never readâ€¦)"), `:259` (`// TODO: files`), `:298-303` (non-number seed silently â†’ 0); `/workspace/bun/docs/runtime/hashing.mdx:170-185`.
evidence: `docs/runtime/hashing.mdx:170` says "The API is the same for each" and the example block on lines 175-176 shows `Bun.hash.adler32("data", 1234)` and `Bun.hash.cityHash32("data", 1234)` - but (verified on 1.4.0) `Bun.hash.adler32("data",1234) === Bun.hash.adler32("data",9999) === Bun.hash.adler32("data")` and likewise for cityHash32: the argument does nothing. `Bun.hash.crc32(" world", Bun.hash.crc32("hello")) === Bun.hash.crc32("hello world")` - a running state, not a seed. `Bun.hash("data","999999") === Bun.hash("data", 0)` (string seed silently dropped). `Bun.hash(Bun.file("/tmp/x")) === Bun.hash("")` - file contents are never read (the `// TODO: files` is 2+ years old; issues #19574 and #31720 both ask for incremental/file input).
why bad: An API that accepts and silently discards an argument, in a family documented as uniform, is a correctness trap. A user who expected `adler32(data, seed)` to randomize gets a fixed, predictable hash. The `Bun.file()` case returns a confidently-wrong answer with no error.
bun 2.0 proposal: Throw on an extra seed argument for seedless algorithms; throw (or actually read) for file-backed Blobs; rename/document `crc32`'s second parameter as the running value (as Node's `zlib.crc32(data[, value])` does).
blast radius: low - the changed cases are all currently no-ops or wrong answers.
confidence: high.

### `Bun.CryptoHasher.digest(encoding)` accepts lossy text encodings, and `digest(hashInto)` demands a 64-byte buffer for a 16-byte hash
what: `DigestEncoding` includes `"utf8" | "ucs2" | "utf16le" | "latin1" | "ascii"` - decoding raw digest bytes as text silently corrupts them. Separately, the `hashInto` overload requires the TypedArray be â‰Ą `EVP_MAX_MD_SIZE` (64) regardless of the algorithm's `byteLength`, and its three sibling APIs each impose a *different* minimum.
where: `/workspace/bun/packages/bun-types/bun.d.ts:4740` (`DigestEncoding`); `/workspace/bun/src/runtime/node/types.rs:643-655`; `/workspace/bun/src/runtime/crypto/CryptoHasher.rs:709` ("TypedArray must be at least {} bytes").
evidence: Verified on 1.4.0: `new Bun.CryptoHasher("sha256").update("hello").digest("utf8").length === 30` - two of the 32 digest bytes are destroyed by UTF-8 decoding. Node's `Hash.digest()` accepts only `BinaryToTextEncoding` (`hex|base64|base64url|binary`) for this reason; issue #14651 is a user discovering the Bun/Node `DigestEncoding` mismatch, and #15781 is a user bitten by `digest("base64")` padding. Verified: `new Bun.CryptoHasher("md5").update("x").digest(new Uint8Array(16))` throws `TypedArray must be at least 64 bytes` even though MD5 is 16 bytes; `Bun.sha("x", new Uint8Array(32))` succeeds (32-byte min); `Bun.SHA1.hash("x", new Uint8Array(20))` succeeds (exact size). The 64-byte error message leaks an internal BoringSSL constant. `digest(hashInto)` also returns the oversized caller array rather than a `byteLength` view.
why bad: Four of the eight accepted encodings silently destroy digest bytes - an output a user will store and can never match again. The `hashInto` contract is keyed to an internal constant, not the algorithm's documented `byteLength`, and is inconsistent across three sibling Bun APIs.
bun 2.0 proposal: Restrict `digest()`'s encoding to `hex | base64 | base64url` (match Node's `BinaryToTextEncoding`). Accept a `hashInto` buffer of exactly `this.byteLength` and return a correctly-sized view.
blast radius: low - anyone using `digest("utf8")` today is already getting corrupt data; the buffer change only loosens a requirement (but the returned-length change is observable).
confidence: high.

### `crypto.timingSafeEqual` bolted onto the standard Web `Crypto` global
what: Bun adds a non-standard `timingSafeEqual` method to the WHATWG/W3C `Crypto` interface (`globalThis.crypto`), duplicating the `node:crypto.timingSafeEqual` it also ships.
where: `/workspace/bun/src/runtime/crypto/crypto.classes.ts:62-70`; `/workspace/bun/packages/bun-types/globals.d.ts:201` (`timingSafeEqual: typeof import("node:crypto").timingSafeEqual;` inside `interface Crypto`).
evidence: The Web Crypto `Crypto` interface (w3c webcrypto spec) defines only `subtle`, `getRandomValues()`, `randomUUID()`. Node's global `crypto` has no `timingSafeEqual`; neither does any browser or Deno. Commit `f649aae36f` (2022-12-12, "[crypto] Implement crypto.timingSafeEqual", "Fixes #1308") shows it was added to satisfy issue #1308, which linked the *Node* docs (`nodejs.org/api/crypto.html#cryptotimingsafeequala-b`) - i.e. the ask was for `node:crypto.timingSafeEqual`, which Bun has since implemented natively.
why bad: It is a Bun-only extension on a standard global, so code written against it is non-portable, and it is 100% redundant with `require("node:crypto").timingSafeEqual`. It exists only because Bun's `node:crypto` was incomplete in 2022.
bun 2.0 proposal: Remove `crypto.timingSafeEqual` from the global `Crypto` object; keep `node:crypto.timingSafeEqual`.
blast radius: low - one-line migration, and TypeScript already points at the Node type.
confidence: high.

### `Bun.CSRF` defaults to the insecure configuration, and its generate/verify signatures disagree
what: `sessionId` and `secret` are both optional, and omitting either produces tokens the docs themselves describe as unsafe/unusable. `generate()` takes the secret positionally; `verify()` takes it inside the options object. Tokens are not self-describing, so `encoding`/`algorithm` must be re-specified on verify or it silently fails.
where: `/workspace/bun/packages/bun-types/bun.d.ts:2199-2283`; `/workspace/bun/src/runtime/api/csrf_jsc.rs:60-72` (`generate` secret positional), `:230-236` (`verify` secret in options); `/workspace/bun/docs/runtime/csrf.mdx:17-21`, `:165`.
evidence: The docs ship a warning callout that admits the default is unsafe - `docs/runtime/csrf.mdx:18`: "Always pass a `sessionId` â€¦ Without it, a token is only bound to the secret - any token the server has ever issued validates for every user". And `csrf.mdx:165`: "tokens won't verify across servers or workers, or after a restart." Verified on 1.4.0: `Bun.CSRF.verify(Bun.CSRF.generate("s"), {secret:"s"})` â†’ `true` for *any* caller (no session binding); a token generated with `encoding:"hex"` verified with the default encoding silently returns `false` (and previously caused an integer overflow - issue #18341); `Bun.CSRF.generate()` succeeds with the ephemeral secret but `Bun.CSRF.generate(undefined)` throws `Secret is required` (`f()` â‰  `f(undefined)`). The API also shipped for months with no documentation (#28142, #28232).
why bad: For a *security* primitive, the zero-argument path is the one the docs tell you never to use - a pit-of-failure default. The positional-vs-options asymmetry between the two halves of the same namespace is a footgun, and the encoding/algorithm-mismatch-returns-false behavior contradicts `Bun.password.verify`, which self-describes via PHC.
bun 2.0 proposal: Make `secret` required and make both functions take it in the same place (options object). Embed the algorithm and encoding in the token so `verify` needs only `{secret, sessionId}`. Strongly consider requiring `sessionId`.
blast radius: medium - requires a signature change, but existing tokens are short-lived (24h default) so there is no persistent-data migration.
confidence: high.

### `Bun.password`'s bcrypt >72-byte SHA-512 pre-hash is a proprietary format that locks users in
what: For passwords longer than 72 bytes with `algorithm: "bcrypt"`, Bun pre-hashes with raw binary SHA-512 before bcrypt, but encodes the result as an ordinary `$2b$` string indistinguishable from a standard bcrypt hash.
where: `/workspace/bun/src/runtime/crypto/PasswordObject.rs:321-336` (hash) and `:388-398` (verify); `/workspace/bun/docs/runtime/hashing.mdx:103`; `/workspace/bun/packages/bun-types/bun.d.ts:3228-3229`.
evidence: `docs/runtime/hashing.mdx:103`: "Instead of silently truncating, `Bun.password.hash` with the `bcrypt` algorithm hashes any password longer than 72 bytes with SHA-512 before passing it to bcrypt." No other bcrypt implementation (bcryptjs, node-bcrypt, Spring, passlib) applies this; passlib's equivalent (`bcrypt_sha256`) uses a *distinct* hash prefix precisely so the two schemes can coexist. Issue #19032 is a user already hitting a `Bun.password.verify`â†”`bcryptjs` mismatch.
why bad: A `$2b$â€¦` hash produced by Bun for a 73-byte password is silently un-verifiable by every other bcrypt library, and a `$2b$â€¦` hash produced by another library for a 73-byte password is silently un-verifiable by Bun (`verify` applies the same pre-hash). There is no marker in the output format, so the corruption is discovered only at migration time, per-user, as a login failure. The intent (don't truncate) is good; the unmarked encoding is the regret.
bun 2.0 proposal: Either tag pre-hashed outputs with a distinct identifier (e.g. a PHC `$bcrypt-sha512$` scheme) so `verify` can detect both forms, or throw on >72-byte bcrypt passwords and let the caller choose. Document that existing >72-byte bcrypt hashes are Bun-specific.
blast radius: medium - only affects the >72-byte corner, but those rows in a real password database are unrecoverable without the new `verify` accepting the old form too.
confidence: high.

### `Bun.randomUUIDv7("hex")` returns a dashed UUID, not hex; `Bun.randomUUIDv5` is not random
what: The `"hex"` encoding of `randomUUIDv7`/`randomUUIDv5` returns the 36-character canonical dashed UUID string, not a hex dump, while `"base64"`/`"base64url"`/`"buffer"` genuinely encode the 16 raw bytes. `randomUUIDv5` (name-based SHA-1) is fully deterministic yet is named `random*`.
where: `/workspace/bun/src/runtime/webcore/Crypto.rs:244-254` (the `Encoding::Hex` branch prints the 36-char dashed form); `/workspace/bun/packages/bun-types/bun.d.ts:9353-9376`.
evidence: Verified on 1.4.0: `Bun.randomUUIDv7("hex")` â†’ `"019f2174-15f5-7000-ba78-e2bf008baf62"` (36 chars, contains `-`); `Bun.randomUUIDv7("base64")` â†’ 24-char base64 of the 16 bytes. The error message the API emits lists `"Encoding must be one of base64, base64url, hex, or buffer"`. `Bun.randomUUIDv5("www.example.com","dns")` returns the identical value on every call (verified).
why bad: `"hex"` is a lie - there is no way to get the actual 32-char hex encoding of the UUID bytes, and `"hex"` is inconsistent with the other three encodings in the same parameter. Naming a deterministic name-based function `randomUUIDv5` misleads users about its security properties.
bun 2.0 proposal: Accept `"hex"` to mean real hex (32 chars, no dashes) and add an explicit default like `"uuid"`/`undefined` for the canonical form; rename `randomUUIDv5` to `uuidv5` (keep an alias through a deprecation window).
blast radius: low for the rename; medium for `"hex"` (anything parsing the 36-char output would break - a deprecation + new value is safer).
confidence: high (behavior), medium (that the team would prioritize it).

### `Bun.password` argon2 hard-codes `parallelism = 1` with no option; no `needsRehash`
what: `Bun.password.hash` exposes `memoryCost` and `timeCost` but silently hard-codes argon2 `parallelism` to 1, and there is no way to ask whether a stored hash uses out-of-date parameters.
where: `/workspace/bun/src/runtime/crypto/PasswordObject.rs:218` ("we don't support the other options right now, but can add them later if someone asks"), `:229-235` (`p: 1` hard-coded), `:142-144`, `:312-315` ("eventually we should do something that about that"), `:294` ("This is purposely simple because nobody asked to make it more complicated"); `/workspace/bun/packages/bun-types/bun.d.ts:3099-3111` (`Argon2Algorithm` has only `memoryCost`/`timeCost`).
evidence: Four verbatim source comments admitting the limitation (quoted above). Issue #9119 ("Bun.password.needsRehash") has been open since 2024 asking for the PHP-style `password_needs_rehash` companion; #19360 asks for scrypt; #9654 asks for a pepper/secret option. `pwhash.rs:158` forces `thread_mode: Sequential` so even `p` values parsed from a foreign PHC hash are computed serially.
why bad: RFC 9106 and OWASP recommend `p â‰Ą 1` tuned to the host; Bun can *verify* a foreign hash with `p=4` but can never *produce* one, so any parameter-migration story is one-way. Without a `needsRehash`, users can't implement the standard rotate-on-login pattern at all. The source explicitly says the design is "purposely simple because nobody asked" - three open issues are people asking.
bun 2.0 proposal: Add `parallelism` to `Argon2Algorithm` and `Bun.password.needsRehash(hash, options)` (small additive surface; only 2.0-relevant because the `Argon2Algorithm` object shape and the "default params" are a compat contract).
blast radius: low - additive.
confidence: medium (these are feature gaps with explicit regret comments, not behavior changes).

## Not included (considered and rejected)
- `Bun.hash` being non-cryptographic at the top-level name: the JSDoc (`bun.d.ts:2301`) and docs already state "This is not a cryptographic hash function"; I found no issue where a user confused `Bun.hash` with a crypto hash. (Issue #32741 is about Bun's *internal* use of fixed-seed wyhash for `trustedDependencies`, not the public API.)
- Argon2 default `m=65536 (64 MiB), t=2, p=1` (`pwhash.rs:84-92`): this is libsodium's `INTERACTIVE` preset and is *stronger* than OWASP's recommendation; issue #8409 ("@node-rs/argon2 is faster than Bun.password?") is explained by parameter differences, not a bad Bun default.
- `Bun.password.verify` auto-detecting the algorithm from the hash: the PHC format is designed for this, `Bun.password` never emits anything else, and the explicit-algorithm override throws on mismatch (verified). Not a regret.
