# Bun 2.0 candidates - focus area: `bun:test`

### mock.module() is process-global with no restore; default single-process run leaks mocks across files

what: `mock.module()` writes into a process-global module-mock registry that is never reset, cannot be undone, and - because `bun test` runs every file in one process by default - leaks into every other test file in the run.
where: `/workspace/bun/src/jsc/bindings/BunPlugin.cpp:507` (`JSMock__jsModuleMock`); `/workspace/bun/docs/test/mocks.mdx:413` ("`mock.restore()` … does not reset modules overridden with `mock.module()`"); `/workspace/bun/docs/test/runtime-behavior.mdx:302-314` ("The test runner runs all tests in a single process by default … Tests share global state"); `--isolate` flag in `/workspace/bun/src/runtime/cli/Arguments.rs`.
evidence: Issue #31316 ("Real-world vitest → bun test migration: gaps that force per-file process isolation"): "`mock.module()` mutates a process-global registry that is never reset between test files, so running multiple files in one `bun test` process leaks mocks across files … **the headline blocker**." Issue #7823 "Restore mock.module using mock.restore not work as expect"; #7376 "Implement the Ability to Revert the Mocking of Modules"; #30242 "captured module namespace is mutated in-place"; #6024 "Make `bun test` isolated"; #9243 "Implement a lightweight version of test isolation". The maintainer closing #12823 wrote: "Fixed by #29354 - `bun test --isolate` … This is currently **experimental** and opt-in. `--isolate` is not the default today but **may become the default in Bun v1.4**."
why bad: The default (`--isolate` off) is the unsafe one: test correctness depends on file ordering, and there is no API to undo a module mock. The team has already written down that the default should flip. #31771 ("`bun test --isolate`: module records retained across per-file global swaps → linear RSS growth") shows the opt-in path isn't production-ready yet either.
bun 2.0 proposal: Make per-file isolation (`--isolate` semantics) the default; add `mock.module.restore(id?)` / `jest.unmock` / `jest.resetModules`; keep a `--no-isolate` escape hatch for the speed-sensitive case.
blast radius: high - changes observable module-graph behaviour for every multi-file suite, but the current behaviour is widely regarded as a bug by users.
confidence: high

### `.resolves` / `.rejects` synchronously block the event loop instead of returning a Promise

what: `expect(promise).resolves.toBe(x)` and `.rejects.*` spin the event loop inside the matcher until the promise settles, so Bun users write them without `await` - the opposite of Jest/Vitest, where the chain returns a Promise that MUST be awaited.
where: `/workspace/bun/src/runtime/test_runner/expect.rs:409` (`global_this.bun_vm().as_mut().wait_for_promise(promise)`; also :852 and :1454); types at `/workspace/bun/packages/bun-types/test.d.ts:928` (`resolves: Matchers<Awaited<T>>` - not `Promise<...>`) and `:936` (`rejects: Matchers<unknown>`).
evidence: Issue #15457 "Undocumented incompatibility between `bun:test` and `jest` with `expect.reject` and `expect.resolves`": "The matchers from `bun:test` will block until the `Promise` is resolved or rejected. The matchers from jest do not block, and instead return `Promise`s that must be awaited." Also #15428 ("`.resolves.toSatisfy(...)` passes in the promise directly") and #19679 ("`expect.hasAssertions` only works if the expect is wrapped inside a Promise").
why bad: (1) A `bun:test`-authored `expect(p).rejects.toThrow()` is a **vacuous no-op** in every other runner (the repo's own review guide calls un-awaited `.rejects` a rejected pattern). (2) Running the event loop synchronously inside a matcher is a re-entrancy hazard: it deadlocks if the promise depends on a timer that `useFakeTimers()` has captured, and its semantics under `test.concurrent` are undefined (two concurrent tests can't both block the loop). (3) The types lie: they say `void`, so TypeScript never tells you an `await` is needed or not.
bun 2.0 proposal: Make `.resolves`/`.rejects` terminal matchers return `Promise<void>` (Jest semantics). A transition lint could warn on un-awaited chains.
blast radius: medium - tests that already `await` (the Jest-idiomatic form) keep working; only tests relying on the blocking shortcut change.
confidence: high

### `jest.mock` / `vi.mock` are aliased to the un-hoisted `mock.module`, and `vitest` / `@jest/globals` imports are silently hijacked

what: Inside `bun test`, `jest.mock`, `vi.mock` and `mock.module` are the SAME un-hoisted function, and the resolver rewrites any `import ... from "@jest/globals"` or `from "vitest"` to `bun:test` - so migrated Jest/Vitest code type-checks and "works" but has different mocking semantics.
where: alias wiring: `/workspace/bun/src/runtime/test_runner/jest.rs:427` (`jest.put(..., b"mock", mock_module_fn)`), `:442` (`vi.put(..., b"mock", mock_module_fn)`); resolver hijack: `/workspace/bun/src/resolve_builtins/HardcodedModule.rs:773-792` (`BUN_TEST_EXTRA_ALIAS_KVS`: `"@jest/globals"` → `"bun:test"`, `"vitest"` → `"bun:test"`); no hoisting: `/workspace/bun/docs/test/mocks.mdx:295-327` ("To make sure a module is mocked before it's imported, use `--preload`").
evidence: Jest hoists `jest.mock()` factories above imports via babel-plugin-jest-hoist; Vitest hoists `vi.mock()`. Bun never hoists - the docs admit it by prescribing `--preload`. Jest's factory-less `jest.mock(path)` (auto-mock) errors in Bun ("mock(module, fn) requires a module and function", `/workspace/bun/src/jsc/bindings/BunPlugin.cpp:518`). Issues: #29834 "Support `jest.mock(module)` in Bun test runner"; #16140 "Implement `vi.mock()`" (open even though `vi.mock` exists - the existing alias is not acceptable); #24185 "Jest automatic mock compatibility"; #11018 "Automatic mock/mocking".
why bad: The names promise Jest/Vitest semantics but deliver something quieter and more dangerous: the real module's side effects run first, and partial factories drop exports in-place (#30242). Silently rewriting `from "vitest"` to `bun:test` is the worst variant - even a user who deliberately `bun install`ed vitest gets Bun's partial `vi` instead.
bun 2.0 proposal: Either hoist `jest.mock`/`vi.mock`/`mock.module` calls in the transpiler (the transform already exists for global injection) or rename the Bun primitive to something that does not claim Jest/Vitest semantics, and stop aliasing the `vitest` package specifier.
blast radius: medium - hoisting is almost always what the author wanted; the rare intentional late-mock can keep using an explicitly-unhoisted spelling.
confidence: high

### `jest.resetAllMocks()` / `vi.resetAllMocks()` are silently the same function as `clearAllMocks()`

what: Bun binds `resetAllMocks` and `clearAllMocks` on both `jest` and `vi` to the identical native `JSMock__jsClearAllMocks`, which only clears call history - it never removes implementations, which is the entire point of `resetAllMocks` in Jest and Vitest.
where: `/workspace/bun/src/runtime/test_runner/jest.rs:430-431` (`jest.put(..., b"clearAllMocks", clear_all_mocks); jest.put(..., b"resetAllMocks", clear_all_mocks);`) and `:445-446` (same for `vi`); `/workspace/bun/src/jsc/bindings/JSMockFunction.cpp:642-664` - the implementation even has a comment admitting it: "seems similar to what we do in JSMock__resetSpies, but we actually only clear calls, context, instances and results" and calls `spyObject->clear()`, not `reset()`.
evidence: The C++ already has the correct `reset()` (`JSMockFunction.cpp:359-365`: `clear()` + `implementation.clear()` + `fallbackImplmentation.clear()` + `tail.clear()`) - it's just not what `resetAllMocks` is bound to. Bun's own docs table at `/workspace/bun/docs/test/mocks.mdx:77` correctly distinguishes per-instance `mockReset()` ("Clears call history and removes implementation") from `mockClear()`, so the alias is contradicted by Bun's own docs.
why bad: A ported Jest/Vitest `afterEach(() => jest.resetAllMocks())` silently stops removing `mockReturnValue`/`mockImplementation` between tests, producing cross-test leakage that is extremely hard to diagnose. This is a correctness lie, not a missing feature.
bun 2.0 proposal: Bind `jest.resetAllMocks`/`vi.resetAllMocks` to a real `resetAll` that calls `reset()` on every active mock. (Arguably fixable in 1.x, but it IS a behaviour change.)
blast radius: low - nobody's test can be correctly depending on `resetAllMocks` doing less than it claims.
confidence: high

### The global `test`/`expect`/`describe` are a transpiler illusion, and importing anything from `bun:test` makes them all vanish

what: Bun "injects" 14 globals (`test`, `it`, `describe`, `expect`, `expectTypeOf`, `beforeAll`, `beforeEach`, `afterEach`, `afterAll`, `jest`, `vi`, `xit`, `xtest`, `xdescribe`) by declaring them as unbound module-scope symbols in the parser and prepending an `import {...} from "bun:test"` - but the entire injection is skipped if the file already imports *anything* from `bun:test`, `@jest/globals`, or `vitest`, and the names are never put on `globalThis`.
where: symbol declaration: `/workspace/bun/src/js_parser/p.rs:3161-3187`; the skip-on-any-import bailout: `/workspace/bun/src/js_parser/parse/parse_entry.rs:1888-1909` (`for item in p.import_records.items() { if item.path.text == b"bun:test" || ... { break 'outer; } }`); the injected name table: `/workspace/bun/src/js_parser/parser.rs:1956-1971` (`Jest::FIELDS`). Types are deliberately NOT global: `/workspace/bun/packages/bun-types/test-globals.d.ts:1` ("Do not include this file in ./index.d.ts") and `/workspace/bun/packages/bun-types/index.d.ts` references `test.d.ts` but not `test-globals.d.ts`.
evidence: Issue #4007 (open since 2023): "Importing anything from bun:test requires all test globals to be explicitly imported" - `import { mock } from "bun:test"` then a bare `test(...)` throws `ReferenceError: Can't find variable: test`, because `mock`/`spyOn`/`setSystemTime`/`onTestFinished`/`setDefaultTimeout` are NOT in `Jest::FIELDS` but the one import kills injection of everything that IS. Issue #4931 "ReferenceError: Can't find variable: jest" (no real global). Issue #12034 "Inject bun:test globals in imported files as well".
why bad: This is the worst possible combination: the runtime magic teaches people to rely on globals, the type system says the globals don't exist, `globalThis.test`/`typeof expect` behave inconsistently with real globals, and the whole feature inverts the moment you need `mock`. Vitest made globals opt-in (`globals: true`); Node's `node:test` requires imports; Jest uses real globals. Bun chose a fourth option that has the costs of all three.
bun 2.0 proposal: Pick one: (a) make them real, typed `globalThis` properties in test files (and never remove them just because of an import), or (b) drop injection and require imports. Either way add `mock`, `spyOn`, `setSystemTime`, `onTestFinished`, `setDefaultTimeout` to whichever set exists - the current split is the footgun.
blast radius: high - every `bun test` file relies on this today, but (a) is backwards-compatible for user code.
confidence: high

### `mock.restore()` - inconsistent name, wrong scope, and it doesn't restore what its siblings imply

what: The `mock` namespace has `mock.restore()` (singular, no "All"/"Mocks") which restores every spy/mock, and `mock.clearAllMocks()` (plural) which clears them - and `mock.restore()` does NOT restore `mock.module()` mocks, which live right next to it on the same object.
where: `/workspace/bun/src/runtime/test_runner/jest.rs:420-423` (`mock_fn.put(..., b"module", ...); mock_fn.put(..., b"restore", restore_all_mocks); mock_fn.put(..., b"clearAllMocks", clear_all_mocks);`) - the identical native `JSMock__jsRestoreAllMocks` is separately exposed as `jest.restoreAllMocks` and `vi.restoreAllMocks` (jest.rs:429,444). Types: `/workspace/bun/packages/bun-types/test.d.ts:50-59`.
evidence: `/workspace/bun/docs/test/mocks.mdx:413`: "`mock.restore()` restores every mock at once … It does not reset modules overridden with `mock.module()`." Issue #7823 "Restore mock.module using mock.restore not work as expect"; #6040 "`mock` and `spyOn` are not reset after each test". There is also no `mock.resetAllMocks()` even though `jest.resetAllMocks` and `vi.resetAllMocks` exist.
why bad: Three names (`mock.restore`, `jest.restoreAllMocks`, `vi.restoreAllMocks`) for the same function, an inconsistent pluralization convention on the same object, and the intuitive reading (`mock.restore()` restores everything `mock.*` created, including `mock.module`) is wrong. There is also no `[test] restoreMocks` / `clearMocks` config (Jest/Vitest both have one) - #31316 point 6, #6040.
bun 2.0 proposal: `mock.restoreAll()` / `mock.clearAll()` / `mock.resetAll()`, all covering module mocks; deprecate `mock.restore`; add `[test] restoreMocks = true` to bunfig.
blast radius: low - pure additions plus a deprecation alias.
confidence: high

### `.only` semantics have churned and now throw in CI; `--only` and docs disagree with reality

what: `test.only` / `describe.only` throw at call time when `CI` is detected ("`.only` is disabled in CI environments … set the environment variable CI=false."), and the flagship docs still describe the old semantics where `.only` is a no-op unless you also pass `bun test --only`.
where: the CI throw: `/workspace/bun/src/runtime/test_runner/ScopeFunctions.rs:501-502` (`if cfg.self_only { error_in_ci(global, b".only")?; }`) and `:511-519` (`error_in_ci`). Stale docs: `/workspace/bun/docs/test/writing-tests.mdx:189-200` ("The following command runs only tests #2 and #3: `bun test --only`. The following command runs tests #1, #2 and #3: `bun test`"). Actual within-file narrowing: `/workspace/bun/src/runtime/test_runner/Order.rs:101-103` and `/workspace/bun/src/runtime/test_runner/bun_test.rs:1763-1767` ("note that we overwrite '.yes' with '.contains' to support only-inside-only").
evidence: The behaviour has flipped repeatedly: #5893 "bun test support .only" → #6898 "Make test.only available without a flag" (closed as dup of #7701) → #18189 "Running bun tests with .only doesn't actually run" → #23380 "test.only stops working on 1.2.23" → #16611 "warn when tests are skipped because of `test.only()`".
why bad: A modifier whose behaviour depends on a CLI flag, the `CI` env var, AND the Bun version is not an API - it's a surprise. Throwing on `CI=true` is a reasonable safety opinion, but it's a hard, unannounced break (a test file with a committed `.only` that passed locally turns into a thrown exception in CI). Jest, Vitest, and `node:test` all treat `.only` identically everywhere.
bun 2.0 proposal: Make `.only` unconditionally narrow the run (as it does now within a file), delete the `--only` flag, and downgrade the CI behaviour to a failing WARNING/exit-code (so the committed `.only` is caught without producing a `TypeError` at describe time). Fix the docs in the same change.
blast radius: low - the CI throw is new and the `--only` flag is rarely used deliberately.
confidence: medium (the throw is HEAD behaviour; I could not date the PR that introduced it).

### `test.todo(name, fn)` + `--todo` duplicates `test.failing(name, fn)`

what: Bun's `test.todo` accepts a callback (Jest's errors on one), only runs it when `bun test --todo` is passed, and then reports a passing body as a failure - which is almost exactly what `test.failing` does, unconditionally.
where: `/workspace/bun/packages/bun-types/test.d.ts:502-519`. The `failing` JSDoc at :516-517 admits the overlap verbatim: "`test.failing` is similar to {@link test.todo} except that it always runs, regardless of the `--todo` flag." Docs: `/workspace/bun/docs/test/writing-tests.mdx:137-165` (`test.todo` + `--todo`) and `:241-260` (`test.failing`).
evidence: the quoted JSDoc; Jest's own `test.todo(name)` throws if a second argument is passed, so Bun's callback-accepting `test.todo` is already non-Jest. `test.failing` was added later specifically to match Jest/Vitest (`test.fails` in Vitest), leaving two overlapping mechanisms plus `todoIf`/`failingIf` variants of both.
why bad: Two names for "I expect this body to fail", differentiated only by an obscure CLI flag, means real suites mix both and get different behaviour in CI vs locally. It also blocks Bun from ever adopting Jest's strict `test.todo` (reject a callback).
bun 2.0 proposal: Make `test.todo(name)` a pure marker (no callback, never runs - the Jest contract), remove `--todo`, and steer callback-carrying cases to `test.failing`. Provide a codemod.
blast radius: medium - `test.todo(name, fn)` is common in suites that migrated from an early Bun.
confidence: high

### Test file discovery is four hardcoded suffixes with no way to customize, and positional args are bare substrings

what: `bun test` finds files by exactly `*.test.*`, `*_test.*`, `*.spec.*`, `*_spec.*` over JS-like extensions - not configurable - and a positional argument is a plain substring match against the relative path, not a glob or a path.
where: the suffix table: `/workspace/bun/src/runtime/cli/test/Scanner.rs:460` (`TEST_NAME_SUFFIXES: [&[u8]; 4] = [b".test", b"_test", b".spec", b"_spec"]`); suffix check: `:274-290`; substring filter: `:306-318` (`strings::index_of(name, filter_name)`); explicit-path bypass of the suffix requirement: `:153` / `:360-364` (`could_be_test_file::<false>`). Docs: `/workspace/bun/docs/test/discovery.mdx:12-15` and `:35` ("Filters are substring matches, not glob patterns.").
evidence: Issue #3440 "Allow custom patterns for `bun test`" - 41 👍, open since 2023. The `_test` / `_spec` suffixes are a Go/Zig convention, not a JS one, and Jest's default `__tests__/` directory convention is NOT supported at all. Bun's docs already had to grow `pathIgnorePatterns` as a patch because the pattern set can't be changed (`/workspace/bun/docs/test/configuration.mdx:88-113`).
why bad: You can subtract (`pathIgnorePatterns`) but never add or change, so any repo with a different convention simply can't use `bun test` without renaming files. Substring filters are also surprising: `bun test api` matches a test in `capital.test.ts`. And passing an explicit `./path/to/file.ts` bypasses the suffix check entirely, so the same file is a "test file" from one invocation and invisible from another.
bun 2.0 proposal: Add `[test] include = ["**/*.test.{ts,tsx}"]` (glob) as the one source of truth, deprecate the baked-in `_test`/`_spec` suffixes, and make positional args globs.
blast radius: medium - `_test`/`_spec` users exist, but a one-line bunfig restores them.
confidence: high

### `vi` is a vitest lookalike that is nowhere near Vitest, and the real `vitest` package is shadowed

what: `bun:test` exports a `vi` object "for migrating tests from Vitest", but it is a small subset (`fn`, `spyOn`, `mock`, 4 clear/reset helpers, the timer fns) while the resolver *also* rewrites any `import from "vitest"` to `bun:test` - so you cannot opt out and use real Vitest APIs inside `bun test`.
where: the `vi` object: `/workspace/bun/src/runtime/test_runner/jest.rs:440-447`; the resolver rewrite: `/workspace/bun/src/resolve_builtins/HardcodedModule.rs:783-791`; types: `/workspace/bun/packages/bun-types/test.d.ts:166-199` ("Vitest-compatible mocking utilities, for migrating tests from Vitest to Bun.").
evidence: Issue #31316 item 7 lists what a real migration needs and Bun lacks: "`hoisted`, `importActual`, `stubEnv`/`unstubAllEnvs`, `stubGlobal`/`unstubAllGlobals`, `advanceTimersByTimeAsync`, `waitFor`, `mocked`, `doMock`, `resetModules`". #16140 "Implement `vi.mock()`"; #16142 "Implement `vi.useFakeTimers()`"; #10275 "Implement `vi.useFakeTimers`, `vi.setSystemTime`, `vi.useRealTimers` like vitest".
why bad: `jest.resetModules`, `jest.requireActual`, `jest.unmock`, `jest.doMock`, `jest.isolateModules`, `vi.resetModules`, `vi.importActual` exist in ZERO places in src (verified by grep) - yet `/workspace/bun/docs/test/runtime-behavior.mdx:340` ships a code sample calling `jest.resetModules()`. The `vi` object and the `vitest` alias together create a compatibility claim Bun does not honor, and because of the alias, installing real Vitest side-by-side inside `bun test` is impossible.
bun 2.0 proposal: Remove the `"vitest"` → `bun:test` resolver alias (keep `@jest/globals`, which is a type-only package). Either finish `vi` or mark the unimplemented members so they throw a clear "not implemented" instead of silently doing something different.
blast radius: low for the alias removal; medium if the partial `vi` members start throwing.
confidence: high

### `expect()` has absorbed ~35 non-Jest `jest-extended` matchers into the core namespace, including a zero-argument `expect()` overload

what: `toBeOdd`, `toBeEven`, `toBeNil`, `toBeEmptyObject`, `toContainKey(s)`, `toContainValue(s)`, `toContainAllKeys`, `toContainAnyValues`, `toBeOneOf`, `toInclude`, `toIncludeRepeated`, `toStartWith`, `toEndWith`, `toEqualIgnoringWhitespace`, `toBeWithin`, `toSatisfy`, `toBeValidDate`, `toBeArrayOfSize`, `toBeTypeOf`, `pass()`, `fail()`, … are all baked into Bun's built-in `Matchers<T>` - and `pass`/`fail` forced a bizarre `expect()` zero-argument overload into the callable type.
where: `/workspace/bun/packages/bun-types/test.d.ts:632` (`(actual?: never, customFailMessage?: string): Matchers<undefined>;` - exists only so `expect().pass()` works) and `:937-963` (`pass`/`fail`, `@link https://jest-extended.jestcommunity.dev/...`). Full list grep: 70+ `to*` matchers in `test.d.ts`.
evidence: The JSDoc links every one of these to `jest-extended.jestcommunity.dev`, not to Jest - Bun vendored a third-party matcher library into the core API. `expect.unreachable()` (`test.d.ts:713`), `expect.resolvesTo` / `expect.rejectsTo` (`test.d.ts:653,663`) and `expect(value, customFailMessage)` are further Bun-only surface.
why bad: Every one of these is now a permanent compatibility commitment with no `@deprecated` escape, and they collide with the real `jest-extended` package if a user `expect.extend()`s it (whose implementations have diverged from Bun's snapshot of them). `expect().pass()` / `expect().not.pass()` are semantically nonsensical and exist only because the overload was added. None of this is removable without a major.
bun 2.0 proposal: Move the jest-extended set behind an explicit `import "bun:test/extended"` (or a `[test] extendedMatchers` flag) so the core `expect` is the Jest surface; delete the 0-arg `expect()` overload, `pass`, `fail`, `.not.pass()`.
blast radius: medium - these matchers are used in the wild, but a one-line opt-in import covers it.
confidence: medium (the matchers are clearly non-standard; whether the team regrets them is an inference).

blast radius: low - additive.
confidence: high (docs/code contradiction is verifiable); medium on whether the team considers the `Date`-not-replaced behaviour final.

### A preload file's `beforeAll`/`afterAll` silently become *global* setup - one name, three different scopes

what: `beforeAll` means "before this describe's tests" inside a `describe`, "before this file's tests" at file top level, and "before EVERY test in EVERY file" when the file is loaded via `--preload` - the scope is inferred from where the call happens, and there is no `globalSetup`/`globalTeardown` API.
where: `/workspace/bun/docs/test/lifecycle.mdx:111-135` ("## Global Setup and Teardown - … define the hooks in a separate file. Then use `--preload`"); the `is_in_preload` switch: `/workspace/bun/src/runtime/test_runner/bun_test.rs:98` and `/workspace/bun/src/runtime/test_runner/jest.rs:469`.
evidence: Jest (`globalSetup`/`globalTeardown`/`setupFiles`/`setupFilesAfterEach`) and Vitest (`globalSetup`/`setupFiles`) give each of these scopes its own explicit name. Bun overloads the per-file names. Related lifecycle surprises that users file as bugs are downstream of the implicit scoping: #21177 "`beforeAll` hook runs outside its describe block with test filtering", #19758 "`beforeAll` in nested `describe` runs too early", #12250 "`--bail` and `afterAll` hook".
why bad: `/workspace/bun/docs/test/lifecycle.mdx:286`: "If a `beforeAll` hook throws, every test in its scope is skipped" - for a preload-registered `beforeAll`, that "scope" is the entire multi-file run. A helper that someone moves from a test file into the preload array changes its blast radius from one file to the whole suite with zero signal.
bun 2.0 proposal: Add explicit `[test] globalSetup = "./setup.ts"` (a module exporting `setup`/`teardown`) and make `beforeAll` in a preload file a hard error with a pointer to it, à la Vitest.
blast radius: low - `--preload` + `beforeAll` is documented but uncommon.

### Snapshot tooling has no obsolete-snapshot pruning, and its CI behaviour is undocumented env magic

what: `--update-snapshots` rewrites changed/new snapshots but never removes obsolete ones, and - separately - snapshot *creation* becomes a hard error whenever the `CI` env var is detected, a behaviour nowhere in the docs.
where: the CI gate: `/workspace/bun/src/runtime/test_runner/snapshot.rs:168-175` ("Prevent snapshot creation in CI environments unless --update-snapshots is used" → `err!("SnapshotCreationNotAllowedInCI")`). Missing pruning is admitted by the docs themselves: `/workspace/bun/docs/test/snapshots.mdx:364-371` ("Bun warns about unused snapshots … Remove unused snapshots by deleting them from the snapshot files or by running tests with **cleanup flags if available**.").
evidence: Jest prunes obsolete snapshots on `-u` and fails on them with `--ci`. Bun only warns and the docs hedge with "if available" (there is no such flag). Issues: #12114 "On --update-snapshots, snapshot file remains empty in afterAll()"; #19280 "Enable test/framework to check if `--update-snapshots` has been provided" (no API); #23705 "Snapshots are broken with --rerun-each".
why bad: Without pruning, renamed/deleted tests leave permanent green zombie entries that a later test with the same name will silently match. The undocumented `CI`-keyed flip in behaviour (same command, different result depending on an env var) is exactly the class of magic default the rest of the report covers.
bun 2.0 proposal: `--update-snapshots` prunes obsolete entries (Jest `-u` semantics); document the CI gate and expose an explicit `--ci` flag so the env sniffing is an override, not the mechanism.
blast radius: low.
confidence: high for the CI gate and the missing pruning (code + docs); medium that pruning-on-update is the desired default.
