# Bun 2.0 candidates - Bun.Transpiler & HTMLRewriter

All findings were reproduced against the in-tree debug/canary build or verified in source. Repro
snippets run with plain `bun` (system Bun 1.4.0-canary) on linux-x64.

### HTMLRewriter async handlers block and re-enter the full event loop (Zalgo)

what: When any handler returns a Promise, `HTMLRewriter` calls `vm.waitForPromise`, which spins the entire event loop (timers, I/O, other promise jobs) inside what is otherwise a synchronous `transform()` call.
where: `src/runtime/api/html_rewriter.rs:1309-1316` (`vm().wait_for_promise(promise)` inside `handler_callback`); `src/jsc/event_loop.rs:921-937` (`wait_for_promise` loops `tick()` + `auto_tick()`); `src/runtime/api/html_rewriter.rs:640-642`; `docs/runtime/html-rewriter.mdx:116-125`; `packages/bun-types/html-rewriter.d.ts:3-5` (`element?(element): void | Promise<void>`).
evidence: Source comment at `html_rewriter.rs:640`: `// Since we're still using vm.waitForPromise, we have to also override the error rejection handler.` - "still using" is an in-code admission. Docs admit it: "Async operations block the transformation until they complete" (html-rewriter.mdx:116). Verified repro: `setTimeout(()=>ticked++)` and `queueMicrotask(()=>ticked+=100)` scheduled *before* a synchronous `rewriter.transform("<p>a</p>")` whose handler does `await Bun.sleep(20)` - both callbacks fired *inside* the `transform()` call and the returned string was `"<p>ticks=101</p>"`. This design is also why the API has produced a long series of memory-safety bugs from JS re-entering lol-html borrows: commits `0561f87d42` (UAF when handler rejects during `end()`), `4c8a33ebdb` (detach attribute iterators), `e5e9734c02` (crash when handlers throw), `35e9f3d4a2` (TextChunk null-ptr crash); open issue #31804 (protected Exception roots leak on thrown handler); #15852, #16357 (crashes on throw).
why bad: A synchronous method that runs arbitrary unrelated timers/IO callbacks re-entrantly is the classic "release Zalgo" anti-pattern: user state can be mutated out from under the caller with no `await` in sight. It is also the structural cause of HTMLRewriter's recurring UAF class (JS re-enters while lol-html has lent a `&mut`). Cloudflare avoids this because its `transform()` result is consumed through a real async stream, so awaiting the handler promise never blocks.
bun 2.0 proposal: Make the result consumption async. Either (a) `transform()` returns `Promise<Response>`/`Promise<string>` and handler promises are awaited on the real microtask queue, or (b) keep the sync string overload but throw if a handler returns a Promise. Never call `wait_for_promise` from user-facing sync paths.
blast radius: high - every user with `async element(){}` handlers relies on the current blocking behavior; changing the return type is signature-breaking.
confidence: high.

### HTMLRewriter.transform() advertises inputs it rejects (Blob, BunFile, ReadableStream-backed Response)

what: The docs and `.d.ts` say `transform()` accepts `Blob`, `Bun.file(...)`, and any `Response`, but Blob/BunFile throw `TypeError: Expected Response or Body` and a `Response` backed by a JS `ReadableStream` throws `ERR_STREAM_CANNOT_PIPE` before any handler runs.
where: `src/runtime/api/html_rewriter.rs:438-444` (the type switch: only `is_string()` and `is_typed_array_or_array_buffer()` are handled; Blob falls to `Other`) and `:510` (`"Expected Response or Body"`); `packages/bun-types/html-rewriter.d.ts:169` (`transform(input: Response | Blob | Bun.BufferSource): Response`); `docs/runtime/html-rewriter.mdx:82-92` ("From Blob", "From File" examples) and `:299-303` ("The body is transformed while maintaining streaming capabilities").
evidence: Open issues: #17259 "HTMLRewriter doesn't support reading from a Blob" (the reporter quotes the type signature); #11758 "HTMLRewriter fails to process Responses created from ReadableStreams"; #14216 "HTMLRewriter does not work on ReadableStream + .text()"; #6068. Bun's own test suite admits it: `test/js/workerd/html-rewriter.test.js:947-955` marks the two `ReadableStream`-backed payloads (`"direct"`, `"default"`) as `it.todo`. Reproduced: `new HTMLRewriter().transform(new Blob(["<p>a</p>"]))` and `.transform(Bun.file("/tmp/t.html"))` both throw `TypeError: Expected Response or Body`; `.transform(new Response(new ReadableStream({async start(c){await Bun.sleep(20); c.enqueue(...); c.close()}})))` throws `Error: Failed to pipe stream`.
why bad: Three independent input classes that the docs and types promise are simply not implemented. Worse, "streaming capabilities" depends on the *internal* representation of the body (native HTTP byte streams from `fetch`/`Bun.serve` work; any user-constructed `ReadableStream` does not), which a user cannot reason about.
bun 2.0 proposal: Either implement the full documented surface or narrow it: a 2.0 signature of `transform(input: Response): Response` + `transformSync(input: string | BufferSource): string | ArrayBuffer` would make the contract honest. At minimum, delete the Blob/File/"streaming" claims from docs and types.
confidence: high.

### HTMLRewriter.transform() has a 3-way polymorphic return type with wrong TypeScript overloads

what: `transform()` returns a `Response`, a `string`, or an `ArrayBuffer` depending on the input's runtime type, and the `.d.ts` overload ordering means every non-`Response` binary input is typed wrong.
where: `packages/bun-types/html-rewriter.d.ts:169-181` (three overloads); `src/runtime/api/html_rewriter.rs:404-511` (`transform_`); `docs/runtime/html-rewriter.mdx:73-92`.
evidence: The first overload `transform(input: Response | Blob | Bun.BufferSource): Response` structurally matches `Uint8Array`, `DataView`, and `ArrayBuffer` (all ⊂ `Bun.BufferSource`, `packages/bun-types/bun.d.ts:21`), so TypeScript picks it and reports `Response`. At runtime, all three return an `ArrayBuffer` (verified: `transform(new Uint8Array(...)).constructor.name === "ArrayBuffer"`). The narrower `transform(input: ArrayBuffer): ArrayBuffer` overload at line 181 is unreachable. The string overload was a bolted-on Bun extension: feature request #2325 explicitly offered two designs ("Allow `.transform()` to take a string OR a Response, and it returns its input" **or** "add a second, non-standard method"); Bun picked the polymorphic one. `docs/runtime/html-rewriter.mdx:92` itself flags the divergence: "The Cloudflare Workers implementation of HTMLRewriter only supports `Response` objects."
why bad: One method with three return types keyed on the input's runtime class is un-inferable at call sites that take `Response | string`, and the current `.d.ts` is objectively wrong for every TypedArray/ArrayBuffer input. The extension is Bun-only, so CF-compatibility (the API's whole premise, `html-rewriter.mdx:333`) is already false.
bun 2.0 proposal: Split into `transform(response: Response): Response` (CF-compatible) and a separately-named `transformSync(html: string | BufferSource)` / `transformString(html: string): string`. Fix the overload ordering regardless.
blast radius: medium - the string overload is widely used in Bun SSR code, but a parallel method + deprecation is cheap.
confidence: high.

### HTMLRewriter is a bare global, not `Bun.HTMLRewriter`

what: A single-vendor, non-standard Cloudflare API is installed on `globalThis` instead of under the `Bun` namespace; `Bun.HTMLRewriter` does not exist.
where: `src/jsc/bindings/ZigGlobalObject.lut.txt:39` (`HTMLRewriter  GlobalObject::m_JSHTMLRewriter  ClassStructure`); `docs/runtime/globals.mdx:40`.
evidence: `docs/runtime/globals.mdx:40` lists HTMLRewriter's "Source" as **`Cloudflare`** - the only entry in the whole globals table with that source (every other row is `Web`, `Node.js`, `Cross-platform`, or `Bun`). Verified: `typeof Bun.HTMLRewriter === "undefined"` while `typeof globalThis.HTMLRewriter === "function"`. Every comparable Bun-specific class (`Bun.Transpiler`, `Bun.Glob`, `Bun.CookieMap`, `Bun.Archive`, ...) lives under `Bun.*`.
why bad: Bun's own documented convention is that Bun-proprietary classes live under `Bun.*`, and globals are reserved for Web/Node standards. `HTMLRewriter` is neither: it is a Cloudflare Workers API that no spec defines. Squatting the global name collides with any future WHATWG/W3C `HTMLRewriter` and with userland polyfills (e.g. `@worker-tools/html-rewriter`), and it is the lone inconsistency in the namespacing story.
bun 2.0 proposal: Expose it as `Bun.HTMLRewriter` (primary) and keep the global as a one-release deprecated alias, or gate the global behind a Workers-compat flag.
blast radius: medium - a one-line alias preserves compat during migration; the pure removal is a clean break.
confidence: high.

### `Bun.Transpiler`'s `deadCodeElimination: true` default silently deletes code - the docs' own example outputs `""`

what: A non-bundling, single-file transform API defaults to dead-code elimination, so an expression statement with no observable side effect is dropped; the exact `.transform("<div>hi!</div>")` example in Bun's docs returns the empty string.
where: `src/runtime/api/JSTranspiler.rs:106` (`dead_code_elimination: true` in `Config::default`); `packages/bun-types/bun.d.ts:2447-2454` (`@experimental ... Enabled by default`); `docs/runtime/transpiler.mdx:76-78`.
evidence: Reproduced verbatim from the docs: `new Bun.Transpiler({loader:"jsx"}).transform("<div>hi!</div>")` → `""`. With `deadCodeElimination: false` → `jsxDEV_…("div", {children:"hi!"}, …)`. Open issue #14789 "Bun.transpiler JSX empty result" is a user hitting exactly this ("it just outputs a semicolon, empty result"). The `.d.ts` JSDoc at `bun.d.ts:2447-2450` labels the knob "**Experimental**" yet it is on by default. `docs/runtime/transpiler.mdx:22-51` shows a `transformSync` "output" block that is also unreachable (it additionally contains a stale `import { __require as require } from "bun:wrap"` line the current implementation never emits, and the body is for a different input).
why bad: esbuild's `transform()` and `tsc.transpileModule()` are semantics-preserving; an optimization that deletes user statements belongs to the bundler, not a "strip the types" API. The wrong default makes the *first thing a user types from the docs* appear broken.
bun 2.0 proposal: Default `deadCodeElimination` to `false` on `Bun.Transpiler` (it stays on in `Bun.build`, where tree-shaking is the point). Fix the doc examples.
blast radius: low - output only gets larger/more faithful; anyone depending on DCE can pass `deadCodeElimination: true`.
confidence: high.

### `Bun.Transpiler` duplicates `Bun.build` with a half-dead, divergent option set

what: `TranspilerOptions` re-implements a large slice of `BuildConfig` (`target`, `define`, `tsconfig`, `minify`, `sourcemap`, `external`, `packages`, `loader`) but several of those options are parsed, validated, and then do nothing, while others that work aren't typed; and the loader is artificially limited to JS-only.
where: `src/runtime/api/JSTranspiler.rs:232-282` (`external` parsed + type-checked), `:434-450` (`sourcemap` parsed + validated against `"inline"|"linked"|"external"|"none"`), `:452-459` (`packages` parsed), `:412-432` (`minify: boolean | {whitespace,syntax,identifiers}` - works), `:286-289` (`"only JavaScript-like loaders supported for now"`); `packages/bun-types/bun.d.ts:2368-2485` (`TranspilerOptions` - has `jsxOptimizationInline` but not `external`/`sourcemap`/`packages`/`minify`); `packages/bun-types/bun.d.ts:2439`; `docs/bundler/esbuild.mdx:12,116`.
evidence: Reproduced: `new Bun.Transpiler({loader:"ts", sourcemap:"inline"})` produces no sourcemap (`out.includes("sourceMappingURL") === false`); open issue #30538 "Bun.Transpiler should optionally emit a sourcemap from transformSync / transform". `external:["react"]` + `packages:"external"` produce byte-identical output to omitting them (no linker exists), yet malformed values throw (`"external must be a string or string[]"` at `JSTranspiler.rs:261`) - the API validates inputs it ignores. `jsxOptimizationInline?: boolean` exists only in the `.d.ts` (`bun.d.ts:2439`); `JSTranspiler.rs` never reads it (the parser feature at `src/js_parser/parser.rs:341` is hard-wired `false`). `minify: {whitespace, identifiers, syntax}` works (verified: identifiers got mangled) but is not in `TranspilerOptions`. Users keep asking for the missing half: #8168 (CJS output), #22516 + #21856 + #15310 (CSS loader), #30538 (sourcemaps), #32919/#30295 (fast-refresh parity with `Bun.build`), #5618 (`jsx: "preserve"`), #14541 (a CLI for it). Bun's own doc positions it as the twin of `esbuild.transform()`: "Always true. Use `Bun.Transpiler` to transpile without bundling." (`docs/bundler/esbuild.mdx:116`).
why bad: Two APIs owning the same concept (transpile options) have drifted: one parses-and-ignores `sourcemap`/`external`/`packages`, has a dead `jsxOptimizationInline`, hides a working `minify`, and rejects every non-JS loader. There is no shared schema, so every new `Bun.build` option re-opens the "does Transpiler have it?" question.
bun 2.0 proposal: Replace the class with a function that shares `Bun.build`'s option schema - `Bun.transpile(code, opts)` / `Bun.transpileSync(code, opts)` (esbuild's `transform`/`transformSync` shape) - and make `sourcemap` actually work or throw `TypeError` when unimplemented. Delete `jsxOptimizationInline`, `external`, `packages` from the single-file surface.
blast radius: medium - `Bun.Transpiler` is widely used in SSR/tooling, but a function wrapper over the same engine is a trivial migration.
confidence: high.

### `new Bun.Transpiler()` defaults the loader to `"jsx"`, so TypeScript input throws

what: With no options, the default loader is `jsx` - plain JavaScript + JSX - so the zero-config constructor cannot transpile TypeScript, which is the API's headline use case.
where: `src/runtime/api/JSTranspiler.rs:93` (`default_loader: Loader::Jsx`); `packages/bun-types/bun.d.ts:2508` (`constructor(options?: TranspilerOptions)`); `docs/runtime/transpiler.mdx:10` (`loader: "tsx", // "js" | "jsx" | "ts" | "tsx"`).
evidence: Reproduced: `new Bun.Transpiler().transformSync("let x: number = 1;")` throws `BuildMessage: Expected ";" but found ":"`. The `.d.ts` JSDoc headline on the class (`bun.d.ts:2488`) is literally "Quickly transpile **TypeScript**, JSX, or JS". Every doc example works around it by passing `loader: "tsx"` explicitly.
why bad: Bun's brand is TS-first everywhere else (file loading, `bun -e`, `bun test`). `jsx` is the one loader choice that handles *neither* TypeScript nor plain JS-with-colons, so the zero-arg constructor is a trap. `tsx` is a strict superset of `jsx` for anything a user would paste in.
bun 2.0 proposal: Default `loader` to `"tsx"`.
blast radius: low - `tsx` accepts all valid `jsx` input; the only observable change is that a few TS-looking sequences (e.g. `<Foo>bar` as a cast) parse differently, which is an edge case no zero-config caller is relying on.
confidence: high.

### `.scan()` and `.scanImports()` silently disagree about `require()` calls

what: The two sibling scanning methods return different import sets for the same input: `scan().imports` omits `require(...)` calls that `scanImports()` reports, and the docs describe the *wrong one* as "less accurate".
where: `src/runtime/api/JSTranspiler.rs:1315-1399` (`scan` → full parse → `named_imports_to_js`) vs `:1678-1812` (`scan_imports` → lexer-only fast path); `docs/runtime/transpiler.mdx:95-191`.
evidence: Reproduced on the docs' own example input (`import React from 'react'; const val = require('./cjs.js'); import('./dyn')`): `scan().imports` → `[react, dynamic-import ./dyn]` (no `./cjs.js`); `scanImports()` → `[react, require-call ./cjs.js, dynamic-import ./dyn]`. The docs contradict themselves: the `.scan()` output block at `transpiler.mdx:118-135` lists a `"remix"` import that is not in the input and omits `./cjs.js`, while two paragraphs later `transpiler.mdx:143` documents `require-call` as a kind `.scan()` produces, and the `.scanImports()` output block at `:174-189` *does* include it. The only stated difference is `transpiler.mdx:153`: `.scanImports()` is "faster ... but marginally less accurate due to its performance optimizations" - in reality the "fast" one reports *more*.
why bad: Two methods that a user would expect to differ only in performance differ in correctness, in the opposite direction from the documentation. There is no reason to expose both.
bun 2.0 proposal: One `scan()` with a consistent result; fold the fast path in as an internal optimization. If both must stay, make `scan().imports` a strict superset and fix the doc.
blast radius: low - `scan`/`scanImports` consumers are build tooling; adding the missing `require-call` records is additive.
confidence: high.

### `transformSync`'s polymorphic 2nd argument is a vestigial macro-context back-channel

what: `transformSync(code, x)` treats `x` as a loader string *or* as an opaque "context object" that gets appended as a hidden extra argument to every macro call, an undocumented mechanism that exists only on the sync method and only for the legacy `macro` remap map.
where: `src/runtime/api/JSTranspiler.rs:1498-1533` (`js_ctx_value` capture: `if arg.is_object() { js_ctx_value = arg }` inside the loader slot) and `:1565` (`MacroJSCtx(js_ctx_value.0 as i64)`); `packages/bun-types/bun.d.ts:2521-2528` (`transformSync(code, loader, ctx: object)` / `transformSync(code, ctx: object)`); `docs/runtime/transpiler.mdx` (never mentions `ctx`); `docs/bundler/macros.mdx` (never mentions `ctx`).
evidence: Verified: `t.transformSync(code, "ts", {hello:"world"})` delivers `{hello:"world"}` as an extra trailing argument to the macro function. The `.d.ts` one-line description is "An object to pass to macros" (`bun.d.ts:2526`) with no further docs anywhere. The async `transform()` has no such parameter. The whole thing only matters for the second, legacy macro system - the `macro?: MacroMap` constructor option (`bun.d.ts:2429`, "Replace an import statement with a macro", example package `bun-macro-relay`) - which predates and duplicates the documented `with { type: "macro" }` import attribute (`docs/bundler/macros.mdx:19`) and is not available on `Bun.build` at all. Issue #851 shows this option has already been renamed once (`macros` → `macro`) and that `platform:` used to be the name for `target:` (today `platform` is silently ignored).
why bad: A 2nd parameter whose *type* switches the meaning (string = loader, object = macro context) is a footgun, and it exists to service a duplicate macro mechanism that the rest of Bun has superseded. Nobody discovering `Bun.Transpiler` from the docs can learn any of this.
bun 2.0 proposal: Remove the `ctx` overloads and the `macro?: MacroMap` option; `Bun.Transpiler` should run macros only via the standard `with { type: "macro" }` import attribute. If the remap capability is still wanted, give it the same name/shape on `Bun.build`.
blast radius: low - no doc has ever advertised it; a repo-wide GitHub code search turns up almost no users.
confidence: high.

### Three overlapping DCE knobs with hidden coupling (`deadCodeElimination` / `treeShaking` / `trimUnusedImports` / `exports`)

what: `Bun.Transpiler` exposes three boolean optimization knobs whose defaults depend on each other and flip implicitly when a fourth, unrelated option (`exports`) is present.
where: `src/runtime/api/JSTranspiler.rs:461-469` (read both), `:630` (`tree_shaking = tree_shaking.unwrap_or_else(|| replacements.count() > 0)` - passing any `exports.eliminate`/`exports.replace` silently enables tree shaking), `:644-645` (`trim_unused_imports` defaults to whatever `tree_shaking` ended up as); `packages/bun-types/bun.d.ts:2433-2454`.
evidence: Source lines above. Issue #13251 "trimUnusedImports does not work" and #12892 "Bun.Transpiler cannot trim unused imports in tsx mode" are users confused by the interaction. None of the three is exposed on `Bun.build` (`BuildConfig` has only `treeShaking`, which `src/bundler/bundle_v2.rs:2782` describes as "set by `Bun.build({treeShaking})` for tests/debugging"; `docs/bundler/esbuild.mdx:172` says "Always true"). `docs/runtime/transpiler.mdx`'s Reference block (lines 197-288) documents `trimUnusedImports` but omits `treeShaking`, `deadCodeElimination`, `logLevel`, `replMode`, `autoImportJSX`, `allowBunRuntime`, and `minify`.
why bad: Three knobs for one concept, with defaults derived from each other *and* from a side effect of an unrelated option, is un-discoverable. The sibling bundler API has none of them.
bun 2.0 proposal: Collapse to the single knob `Bun.build` uses (`treeShaking`) plus `minify`, with independent, documented defaults. Passing `exports` must not change optimization defaults.
blast radius: low - all four are undocumented or under-documented and default-off except `deadCodeElimination`.
confidence: high.

### `Bun.Transpiler.transform()` (async) is documented as a footgun

what: The async method exists solely to push the work to a thread pool, and Bun's own docs tell you not to use it for its nominal purpose.
where: `docs/runtime/transpiler.mdx:81` ("Unless you're transpiling _many_ large files, use `Bun.Transpiler.transformSync`. The threadpool overhead often costs more than the transpilation itself.") and `:91` ("If your code uses a macro, it may spawn a new copy of Bun's JavaScript runtime environment in that new thread."); `src/runtime/api/JSTranspiler.rs:657-898` (the entire `TransformTask`/`AsyncTransformTask` machinery - an unsafe bitwise snapshot of the `Transpiler` struct (`:705-735`) exists only for this method).
evidence: The doc quotes above. The async variant also has less capability than the sync one: it cannot take the macro `ctx` argument, and `docs/runtime/transpiler.mdx:36-51` - the only output example for the async path - is stale (shows a `bun:wrap` import and a result for a different input).
why bad: An async method whose maintainers document as slower than its sync twin in the common case buys nothing; it exists because esbuild has one, but esbuild's is faster because esbuild's sync variant pays a process-spawn cost - a constraint Bun doesn't have. The ~250-line `TransformTask` (`ManuallyDrop` bitwise struct copy, raw pointer re-aiming) is a standing memory-safety liability funding a near-useless API.
bun 2.0 proposal: Drop `.transform()` or make it a trivial `Promise.resolve(transformSync(...))` wrapper; real parallelism should go through `Bun.build` or Workers.
blast radius: low - a one-line JS shim preserves behavior for the rare caller.
confidence: medium - the perf claim is the Bun team's own, but I did not re-benchmark it.
