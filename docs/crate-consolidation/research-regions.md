### Open questions
- `src/bun_alloc/lib.rs:3497` `trait Allocator` + `dyn Allocator` appears to coexist with `StdAllocator{ptr,vtable}` as a second allocator-polymorphism system; unclear if one is being phased out.
- `src/ptr/owned.rs` / `src/ptr/shared.rs` are marked OBSOLETE (lib.rs:32) but `owned::alloc_dupe_slice` still has 2 callers (`runtime/test_runner/diff/diff_match_patch.rs:1372`, `s3_signing/credentials.rs:923`); unclear if those should move to `bun_alloc::default_dupe`.
- Whether the 2 `link_interface!` in `bun_core` (`OutputSink`, `ErrnoNames`) survive consolidation depends on whether `bun_sys`+`bun_errno` end up above or alongside the merged foundation crate — if `sys`/`errno` merge into the foundation, both interfaces disappear.

### The sys / io / spawn split — concrete dependency reasons

**Why `io` can't be in `sys`:** `io` depends on four crates that themselves depend on `sys`:

1. `bun_uws_sys` (the hard one) — `src/io/PipeReader.rs:16`, `src/io/lib.rs:92,121,250`: `bun_io::Loop` _is_ `bun_uws_sys::Loop` on POSIX; `EventLoopCtx::platform_event_loop_ptr() -> *mut bun_uws_sys::Loop`. And `uws_sys → {boringssl_sys, http_types, libuv_sys}`. Merging would put the TLS/HTTP stack below syscalls.
2. `bun_threading` — `src/io/lib.rs:1136,1272,1280,2248` uses `UnboundedQueue<Request>`, `WorkPool`/`OwnedTask` (Closer), `Mutex` (source.rs:460). `threading → sys` (Futex.rs:131,189,308 uses `bun_sys::{windows,darwin,linux}`).
3. `bun_collections` — `src/io/posix_event_loop.rs:1407` `hive_array::Fallback<FilePoll,N>`, `TaggedPointer`.
4. `bun_spawn_sys` — `src/io/ParentDeathWatchdog.rs:242` calls `bun_spawn_sys::pdeathsig::set_default(true)` (single call site). `spawn_sys → {sys, analytics}`.

`io` is not "system I/O"; per its own header (`src/io/lib.rs:1-4`) it's "the barely used epoll/kqueue event loop, only used by Bun.write() and Bun.file().text()". It's really the **fd-poll / pipe-reader layer that glues uSockets' loop to runtime owners**. The name is the confusion.

**Why `spawn_sys` is split from `spawn`:** `spawn → io` (process.rs:17 uses `FilePoll`/`KeepAlive`; static_pipe_writer.rs:7 uses `BufferedWriter`) and `io → spawn_sys` (one call: `pdeathsig::set_default`). Putting spawn's posix_spawn wrapper in spawn would make `io → spawn → io`. The split exists for **one boolean setter** plus the fact that `spawn → {event_loop, crash_handler, which}` while `io` must stay below event_loop.

**Why `spawn` is split from `io`:** `spawn → event_loop → io`. `spawn/process.rs:978-980` uses `EventLoopHandle`, `AnyTaskWithExtraContext`, `ConcurrentTask`, `TaskTag` from event_loop. io can't see those (event_loop → io).

**What `threading` provides beyond `std::sync`:**

- `Futex.rs:547` — raw futex/`__ulock_wait`/`WaitOnAddress` wrappers; std hides these.
- `Mutex.rs:412` — SRWLOCK/`os_unfair_lock`/futex; **no poisoning**, `const fn new()`, `is_held_by_current_thread()` debug check, `pub(crate) srwlock` exposed so `Condition` can `SleepConditionVariableSRW` on it directly.
- `ThreadPool.rs:1957` — kprotty's work-stealing pool with intrusive `Task` nodes + `BUN_THREADPOOL_STATS`; std has no pool.
- `unbounded_queue.rs:345` — intrusive MPSC (`Link<T>` field in node type).
- `work_pool.rs:214` — global singleton `ThreadPool` + `IntrusiveWorkTask`/`OwnedTask` traits.
- `Channel.rs:220`, `RwLock`, `Condition`, `Semaphore`, `ResetEvent`, `WaitGroup`, `Guarded` — Zig-style, non-poisoning, some over `LinearFifo` from collections.

### Layering workarounds found

1. **`extern "C"` Rust→Rust upcall shim** — `src/uws_sys/lib.rs:175-348`: opaque `UpgradedDuplex` / `WindowsNamedPipe` with 12+13 `extern "C"` decls (`UpgradedDuplex__ssl_error`, `__close`, `__encode_and_write`…) that resolve at link time to `#[uws_callback(export=…)]` symbols in `src/runtime/socket/UpgradedDuplex.rs:216-654` and `src/runtime/socket/WindowsNamedPipe.rs:523-1416`. Breaks cycle **`bun_uws_sys → bun_runtime → bun_uws_sys`** (uws_sys's `InternalSocket` enum at `src/uws_sys/socket.rs` carries variants for these runtime-owned types). Comment at `lib.rs:175-182` explicitly calls this a "cycle-break shim".

2. **Manual C vtable** — `src/uws_sys/vtable.rs:38-202`: `trait Handler` + `Make::<H>::VT` emits a static `us_socket_vtable_t` of `extern "C"` trampolines per handler type. This one is legitimate (the vtable is consumed by C `loop.c`), not a Rust-layering hack, but coexists with…

3. **Closed-world enum dispatch** — `src/uws_sys/SocketKind.rs:18-60`: 20+ variant `#[repr(u8)]` enum (`BunSocketTcp`, `HttpClient`, `Postgres`, `WsClient`…) stamped on sockets; matched exhaustively in `src/runtime/dispatch.rs`. The low tier declares the tag set, the high tier owns the match — a `bun_dispatch`-style split without using the `dispatch` crate itself. Constraint it works around: uws_sys can't name `bun_http::HTTPContext`, `bun_sql_jsc::Postgres`, etc.

4. **`spawn_sys` split** — `src/spawn_sys/lib.rs:1-20` documents the rationale: carved out of `bun_spawn` so `bun_io` can call `spawn_process_posix` / set `pdeathsig` (`src/io/ParentDeathWatchdog.rs:242`) without the cycle **`bun_io → bun_spawn → bun_event_loop → bun_io`**. Evidence of cycle-breaker state hoisted down: `waiter_thread_flag` (`lib.rs:119-133`) and `pdeathsig` (`lib.rs:144-177`) are AtomicBool globals that `bun_spawn::WaiterThread` and `bun_io::ParentDeathWatchdog` both touch.

5. **`bun_core::spawn_ffi` types-in-core** — `src/bun_core/util.rs:4736`: the `posix_spawn_bun` `#[repr(C)]` request structs (`BunSpawnRequest`, `Action`, `ActionsList`) + extern decl live in `bun_core` so both `bun_core::util::spawn_sync_inherit` and `bun_spawn_sys::posix_spawn` (`src/spawn_sys/posix_spawn.rs:126,474`) share them. A \*\_types-style split folded directly into the foundation crate.

6. **`uws_sys → bun_http_types` dep** — `src/uws_sys/App.rs:6` and `src/uws_sys/h3.rs:505-509` import `bun_http_types::Method::Method` only. This pulls a 5490-LOC crate (→ bun_collections → bun_url) into the sys layer for one enum.

7. **Vestigial wrapper: `bun_uws`** — 1444 LOC, single file, ~90 % re-exports of `bun_uws_sys` (`src/uws/lib.rs:22-35`). Per `src/uws_sys/socket.rs:4-7`: _"THIS IS THE ONE CANONICAL IMPLEMENTATION. `bun_uws` re-exports everything here; do NOT add a parallel `InternalSocket`/`NewSocketHandler` in `bun_uws` again"_. bun_uws adds only `#[uws_callback]` proc-macro re-export (from `bun_jsc_macros`) + a handful of helpers; bun_event_loop/bun_http/bun_install already take `bun_uws` as dep but could take `bun_uws_sys` directly.

### Merge candidates

**Hard constraint on a single `bun_ffi`:** `bun_core` depends on `bun_simdutf_sys` + `bun_windows_sys` (`src/bun_core/Cargo.toml:23,32`); `bun_alloc` depends on `bun_mimalloc_sys`. A unified `bun_ffi` that itself uses `bun_core`/`bun_alloc` would cycle. The \*\_sys layer therefore straddles at least two tiers.

**Proposal — collapse 12 → 2 crates + 3 module merges:**

- **`bun_ffi_leaf`** (`#![no_std]`, zero workspace deps; sits below `bun_alloc`/`bun_core`): absorb `windows_sys` + `simdutf_sys` + `mimalloc_sys` + `brotli_sys` as modules. All four currently have ≤ `bun_opaque` as their only workspace dep. Blocker: `bun_shim_impl` (`src/install/windows-shim/Cargo.toml:18-27`) requires its dep to carry **zero `#[no_mangle]`** symbols; simdutf/mimalloc externs are declarations only (no exports) so this holds, but `brotli_sys` is irrelevant to the shim — feature-gate it or keep `windows_sys` standalone and merge the other three.
  - Alternative with zero new crate: `mimalloc_sys` → module in `bun_alloc`; `simdutf_sys` → module in `bun_alloc` (bun_core already depends on bun_alloc, and every other simdutf_sys importer already transitively has bun_alloc). `windows_sys` stays standalone for the shim.

- **`bun_ffi`** (depends on `bun_core`, `bun_alloc`, `bun_opaque`, `bun_windows_sys`; sits between `bun_core` and `bun_errno`/`bun_sys`): absorb `boringssl_sys` + `libuv_sys` + `zlib_sys` + `libdeflate_sys` + `cares_sys` + `tcc_sys` as modules. Dependency check: their union of deps is {bun_core, bun_alloc, bun_opaque, bun_ptr, bun_windows_sys, bun_wyhash, (self-ref) bun_libuv_sys}. `bun_errno` currently depends on `bun_libuv_sys` (`src/errno/darwin_errno.rs:130` for `UV_E*` constants) — after merge, `bun_errno` would depend on `bun_ffi`, and nothing in that module set depends back on `bun_errno`, so no cycle. `bun_ptr` is used only by `tcc_sys`; either add it as a dep or inline the one use.

- **Fold into sole consumer:**
  - `brotli_sys` → `bun_brotli::sys` (only importer, 654-LOC wrapper).
  - `zlib_sys` → `bun_zlib::sys` (only importer, 1442-LOC wrapper). If done, drop zlib_sys from `bun_ffi` above.
  - `tcc_sys` → `bun_runtime::ffi::tcc` (only importer). If done, drop from `bun_ffi` above.

- **`bun_uws_sys` + `bun_uws` → single `bun_uws`** (net -1 crate, kills the vestigial re-export layer). This does NOT fit in `bun_ffi` because it needs `bun_errno` + `bun_http_types`; it would make `bun_ffi → bun_http_types → bun_collections → bun_core` a very heavy sys crate. Keep as its own crate but drop the wrapper. If `Method` is moved/duplicated down (it's a 10-variant enum), the `bun_http_types` dep disappears and `bun_uws` could join `bun_ffi` — but 9.7 kLOC of socket/response/websocket logic arguably doesn't belong in a "sys" bucket anyway.

- **`bun_spawn_sys`** is mis-named — it's the event-loop-free half of `bun_spawn`, not a vendor binding. Options: (a) rename `bun_spawn_core` and leave as-is; (b) if `bun_spawn` + `bun_io` + `bun_event_loop` merge into one crate in the broader consolidation, the split evaporates entirely (its only reason to exist is the `io → spawn → event_loop → io` triangle). It cannot join `bun_ffi` because it depends on `bun_sys` + `bun_analytics`.

**Net result:** 12 crates → `bun_windows_sys` (kept for shim), `bun_ffi` (~6 modules), `bun_uws`, `bun_spawn_sys` (or dissolved) = 3-4 crates, plus 3 modules absorbed into their wrappers. The `UpgradedDuplex__*` extern-C shim at `src/uws_sys/lib.rs:187-348` survives unless `bun_runtime::socket` and `bun_uws` end up in the same crate.

### Why `install_types` is separate — the cycle

`bun_install` → `bun_resolver` is a hard dep (≥15 use sites for `bun_resolver::fs::FileSystem` / `RealFsTmpfile`: `src/install/lockfile.rs:23`, `PackageManager.rs:1447`, `patch_install.rs:10`, `lockfile/Package.rs:8`, …). `bun_resolver` in turn needs `Dependency`, `Resolution`, `PackageID`, `AutoInstaller` for auto-install (`src/resolver/resolver.rs:20,25,543,874`) and `PackageJSON` field types (`src/resolver/package_json.rs:23`). That would be `install → resolver → install`. `install_types` breaks it by owning the shared value types + the `AutoInstaller` trait object.

**Crates that import `install_types` but NOT `install`:**

- `bun_resolver` — `AutoInstaller`, `Dependency*`, `Resolution`, `Behavior`, `PackageID`, … (`resolver.rs:20-58`, `package_json.rs:23-34`, `result.rs`)
- `bun_options_types` — re-exports `NodeLinker`, `PnpmMatcher` into `schema::api::BunInstall` (`schema.rs:263,320,325-326`)
- `bun_ini` — `NodeLinker`, `npm::Registry::DEFAULT_URL`, `PnpmMatcher`, `create_matcher` (`lib.rs:191,1602,1702,1806`)
- `bun_bunfig` — `NodeLinker::FromExprError` (`bunfig.rs:19`)
- `bun_api` — **vestigial** (Cargo.toml:23 only; zero `.rs` references — it re-exports from `bun_options_types` instead, `src/api/lib.rs:28-30`)

### What `install_jsc` adds

Pure `JSValue`/`CallFrame` host-fn bodies that call into `bun_install`/`bun_ini`. Single consumer (`bun_runtime::dispatch_js2native.rs:25-37`). It cannot fold into `bun_jsc` because it also needs `bun_ast_jsc::log_to_js` (`dependency_jsc.rs:131`, `ini_jsc.rs:111`) and `bun_js_parser_jsc::expr_to_js` (`ini_jsc.rs:220`), both of which depend on `bun_jsc`. It is effectively a `bun_runtime` submodule that was split out.

### Is `install < jsc` intentional?

Yes — it is load-bearing, not accidental. `bun_jsc` uses `bun_install` in three places:

- `src/jsc/AsyncModule.rs:7-8,261-302` — the runtime auto-install `Queue` implements `bun_install::package_manager::run_tasks::RunTasksCallbacks` and calls `vm.package_manager().drain_dependency_list()`.
- `src/jsc/VirtualMachine.rs:3505-3521` — `VirtualMachine::package_manager()` downcasts the resolver's `dyn AutoInstaller` back to concrete `&mut bun_install::PackageManager`.
- `src/jsc/error.rs:76` — `Install(#[from] bun_install::Error)` in `bun_jsc::Error`.

This pins the whole chain `semver ≤ install_types ≤ {options_types, resolver, ini, bunfig} ≤ install ≤ jsc ≤ {install_jsc, semver_jsc, patch_jsc} ≤ runtime`. If `AsyncModule.rs` + `VirtualMachine::package_manager()` moved up to `bun_runtime` (or `install_jsc`), `bun_jsc` would no longer need `bun_install` and every `*_jsc` crate in this region could fold into its base crate.

### Merge candidates

1. **`bun_jsc` + `bun_runtime` (+ every `*_jsc` crate) → one crate.** This eliminates the entire `jsc_hooks.rs` (5378 LOC of fn-ptr plumbing), `__BUN_RUNTIME_HOOKS`/`__BUN_LOADER_HOOKS`/`__BUN_SQL_RUNTIME_HOOKS` vtables, `RuntimeState` opaque-box, `dispatch_js2native.rs`, `link_impl_VmLoaderCtx`, `__bun_blob_from_build_artifact`, `dyn SendQueueOwner`, `ipc_host.rs`, the `#[path]`-mount of `generated_classes_list.rs`, and 5 of the 10 `dispatch.rs` fns (`__bun_run_immediate_task/cancel/wtf_timer/tick_queue/release_task`). Absorbs: `bun_sql_jsc`, `bun_http_jsc`, `bun_install_jsc`, `bun_js_parser_jsc`, `bun_sourcemap_jsc`, `bun_css_jsc`, `bun_bundler_jsc`, `bun_patch_jsc`, `bun_semver_jsc`, `bun_sys_jsc`, `bun_ast_jsc`. **Blocker**: `bun_jsc` is 54k LOC and is a dependency of 11 `*_jsc` crates that `bun_runtime` already depends on — merging requires flattening the whole tier.

2. **`bun_event_loop` + `bun_io` + `bun_spawn` → into the jsc/runtime merge.** Eliminates `__bun_fire_timer`/`__bun_js_timer_epoch`/`__bun_run_file_poll`/`__bun_io_pollable_*`/`__bun_get_vm_ctx`/`__bun_js_vm_get`/`__bun_stdio_blob_store_*`, plus `link_interface! {JsEventLoop, EventLoopCtx, BufferedReaderParentLink, ProcessExit}`. **Blocker**: `bun_install` (80k LOC, no-JSC path) also uses `bun_event_loop`/`bun_io`/`bun_spawn` with `MiniEventLoop` — would drag install into the merge or require keeping `Mini` variants.

3. **`bun_bundler` → into runtime.** Eliminates `link_interface! {DevServerHandle, VmLoaderCtx}`, `extern "Rust" __bun_jsc_generate_cached_bytecode`/`__bun_jsc_enable_hot_module_reloading_for_bundler` (`src/bundler/bundle_v2.rs:1403,1417`). **Blocker**: `bun_install`, `bun_standalone_graph`, `bun_bunfig` depend on `bun_bundler` without JSC.

4. **`bun_dns` → into runtime.** Eliminates `__bun_dns_prefetch`. **Blocker**: `bun_http` and `bun_install` use `bun_dns` without JSC.

5. **`bun_standalone_graph` + `bun_resolver` merge** (or both into runtime). Eliminates `dyn StandaloneModuleGraph` + the raw-ptr downcast at `jsc_hooks.rs:1504`. **Blocker**: none structural — `standalone_graph` already depends on `resolver`; only reason for `dyn` is that `bun_jsc::VirtualMachine` stores it and `bun_jsc` can't depend on `standalone_graph`. If `bun_jsc` merges into runtime, the `dyn` vanishes.

6. **`bun_dispatch` proc-macro crate → delete.** All 10 `link_interface!` sites exist only to break crate cycles. If the declaring crate and all impl crates end up in the same crate, each becomes a plain `enum` with `impl` blocks. No blocker beyond (1)-(3).

7. **Intra-runtime module merges** (no workaround payoff, but compile-time shape): `webview` (973 LOC, leaf) → `api/`; `allocators` (376 LOC) → `webcore/`; `ipc_host.rs` → `api/bun/subprocess/`; `dispatch_js2native.rs` disappears with (1).
