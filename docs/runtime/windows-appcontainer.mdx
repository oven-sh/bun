title: Windows AppContainer
description: What works, what the host must configure, and what the sandbox forbids when running Bun under an AppContainer (lowbox) token.
---

Bun can run inside a Windows AppContainer — the sandbox used by packaged apps and
embedders that launch worker processes with a lowbox token
(`CreateAppContainerProfile` + `PROC_THREAD_ATTRIBUTE_SECURITY_CAPABILITIES`).
Running a script, `bun run`, `bun install` (default linker), `bun build`,
`bun test`, `Bun.spawn`/`child_process` with piped stdout/stderr, IPC
(`process.send`), `Bun.serve` + `fetch` to itself, `fs.watch`, workers,
`bun:sqlite`, and `bun:ffi` all work inside the container.

This page lists what the **host** (the process creating the container) must
configure, and the platform behaviors that no amount of configuration changes.

## Host configuration checklist

The container only touches what its SID (or `ALL APPLICATION PACKAGES`) is
ACL'd for, plus OS directories. Before launching Bun sandboxed:

| Grant                                           | Why                                                                                                                      |
| ----------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------ |
| `RX` on `bun.exe` and its directory             | the loader must read the image                                                                                           |
| `(OI)(CI)` read (or modify) on the project tree | sources, `node_modules`, lockfile                                                                                        |
| a writable working directory                    | Bun reads its cwd at startup                                                                                             |
| nothing for TEMP                                | `CreateProcess` rewrites `TMP`/`TEMP` for an AppContainer child to the package's `AC\Temp`, which exists and is writable |

```sh
icacls C:\path\to\bun-dir /grant "*<container-sid>:(OI)(CI)(RX)" /T
icacls C:\path\to\project /grant "*<container-sid>:(OI)(CI)(M)" /T
```

**Network needs capabilities.** With no capabilities the container has no DNS
and no sockets (`EACCES`/`DNSResolveFailed`). Pass `internetClient` for
outbound internet, `privateNetworkClientServer` for private-range addresses
(corporate DNS included), `internetClientServer` for unsolicited inbound.

**Services and CI must grant the window station.** Outside an interactive
session, add the container SID to the window station and desktop DACLs or any
executable importing `user32.dll` (Bun does) dies at load with
`STATUS_DLL_INIT_FAILED` (0xC0000142). Interactive sessions already carry the
right ACEs.

**Loopback to processes outside the container is blocked by design.**
In-process and same-container loopback always works (`Bun.serve` + `fetch`
to yourself is fine, as is talking to your own child processes). Reaching an
unsandboxed local process's `127.0.0.1` port — a database, a dev server, an
inspector — requires the machine-level debugging exemption
`CheckNetIsolation.exe LoopbackExempt -a -p=<container-sid>` (admin), which
shipped apps cannot rely on.

## Behavior differences inside the container

- **Named pipe servers must live under `\\.\pipe\LOCAL\`.** The OS denies pipe
  creation everywhere else; `net.createServer().listen(path)` and
  `Bun.listen({ unix })` outside that prefix fail. Connecting to an existing
  pipe elsewhere is allowed if that pipe's ACL admits the container. Bun's own
  internal pipes (stdio, IPC, ConPTY) already use `LOCAL\`.
- **True symlinks are impossible** — a lowbox token cannot hold
  `SeCreateSymbolicLinkPrivilege`, so `fs.symlink` with `"file"`/`"dir"` fails
  with `EPERM`. Hardlinks work. **Junctions are created but quarantined** on
  client Windows builds: the kernel marks them as _untrusted mount points_
  whose traversal is denied, so junction-based layouts are unsupported inside
  the container — creation appears to succeed but the junction is unusable.
  (Some Windows Server builds leave such junctions traversable.) Use
  `bun install` with the default (hoisted) linker; the isolated linker
  depends on junctions and is unsupported in sandboxes. `bun link` and
  `bun unlink` are likewise unsupported: the global link directory sits
  outside any container grant and their path resolution is denied.
- **`os.homedir()` points at an inaccessible directory.** `USERPROFILE` is
  inherited unchanged, but the profile directory carries no ACE for the
  container. Drive roots (`C:\`) are not listable either; Bun's resolver
  treats those inaccessible ancestors as opaque directories (a
  `node_modules` physically inside one is not found — keep dependencies
  under the granted project tree). `fs.realpath` fails with `EPERM` inside a
  container: its component walk stats the drive root, and the native variant's
  DOS-name translation opens the mount manager. Bun's own module resolution
  and `bun install` do not go through `fs.realpath` and are unaffected.
- **Process visibility is limited.** `process.kill` of unrelated pids and
  enumerating other users' processes are denied; managing your own children
  works.
- **`"ignore"` stdio fails by default.** It opens the `NUL` device, whose ACL
  predates AppContainers and is re-derived at every boot; it admits the
  container only when the device ACL carries the container SIDs (grant them per
  boot from an elevated context). `"inherit"` stdin works without host setup;
  piped stdout/stderr work.
- A package cache created **outside** the container is currently
  re-validated as a miss inside it; prefer letting the sandboxed process
  populate its own `BUN_INSTALL_CACHE_DIR`.

## Verifying a sandboxed setup

A minimal smoke test that exercises the historically fragile paths:

```js
// smoke.js — run with: bun smoke.js (inside the container)
const { spawnSync } = require("child_process");
const r = spawnSync(process.execPath, ["-e", "0"], { stdio: ["inherit", "pipe", "pipe"] });
console.log("spawn+pipes:", r.status === 0 ? "ok" : r.error);
console.log("resolve:", require.resolve("./smoke.js"));
require("net")
  .createServer(() => {})
  .listen("\\\\.\\pipe\\LOCAL\\smoke-" + process.pid, function () {
    console.log("pipe server: ok");
    this.close();
  });
```

If `spawn+pipes` hangs, the Bun build predates AppContainer support (requires
Bun v1.4.0 or newer).
