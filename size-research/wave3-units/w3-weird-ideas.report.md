#### W1 — `embedded-js-zstd`: per-item zstd-19 the 158 internal-module sources, decompress lazily at `requireId`

- **saving_mb:** **1.306** (linux). Derivation BY ADDRESS: the 158
  `Bun::InternalModuleRegistryConstants::*CodeBytes` `r` symbols in nm-dem.txt sum to
  1,756,925 B (158 unique addresses, span 1,758,159, padding 1,234); their real bytes
  extracted at (addr - 0x211000) from the .rodata section dump and per-item
  zstd-19-compressed with the canary's own `node:zlib` total 387,811 B.
  1,756,925 - 387,811 = 1,369,114 B = **1.3057 MB**.
  NET-NEW over the banked inventory after subsuming row 3: **+0.99 to +1.01 MB**.
- **confidence:** HIGH. Every link in the chain is measured or read from source (F1-F7).
- **risk:** LOW.
  - RSS ledger (honest, the only real cost): a process that requires K internal modules
    trades K modules' worth of file-backed, shared, evictable .rodata for K modules'
    worth of malloc'd anon memory (the ref-counted `StringImpl` buffer). Proportional
    to what you actually load; worst case (all 158) is +1.76 MB anon. This is STRICTLY
    more favorable than the already-A-ranked brotli row 6 (+404 KB, flat, for every
    brotli user) and the already-SHIPPED lshpack patch (786 KB). The compressed blobs
    (0.39 MB) stay in .rodata.
  - Fork-heavy workloads (the test runner) lose page-cache sharing of the loaded
    modules' source across sibling processes. Same class as row 6/lshpack.
  - `.toString()` of builtin-module functions returns IDENTICAL bytes — unlike EJ1,
    this has ZERO observable behavior change and ZERO JSC-builtin-parser risk (EJ1's
    skeptic found 7 modules crash JSC's `ASSERT_ENABLED` validator when minified;
    compression cannot trigger that because the text is unchanged).
  - Optional extra-conservative variant (exactly Dylan's ICU `keep-raw.txt` design): a
    per-module exclusion list for anything the maintainers deem latency-critical. Not
    needed by the data, but it is the same author's own mitigation pattern.
- **perf:** **one-time-lazy(`InternalModuleRegistry::requireId`, InternalModuleRegistry.cpp:162-169)**.
  The guard `if (!value || value.isUndefined())` already makes the body run exactly once
  per (module, VM). The added ZSTD_decompress is a MEASURED 0.09%-3.9% of the cold
  `require()` it precedes (F4). ZERO modules are evaluated on any unconditional startup
  path (F3) so `bun --version` / `bun -e ''` gain 0 ns. Cite-the-upstream-argument:
  this IS the mechanism of GT#1's blessed ICU design and of
  `patches/lshpack/bss-huff-tables.patch`, both already shipped by the same maintainers.
  A plausible IMPROVEMENT component: 1.37 MB less file-backed .rodata to download, mmap,
  and demand-page.
- **regression:** none user-observable. Honest ledger: the per-loaded-module RSS shift
  above.
- **windows:** YES, ~**1.30 MB** (same `bundle-modules.ts`, same `InternalModuleRegistry.cpp`,
  same linked zstd; PE `.rdata` holds the same 158 arrays). Windows only needs 2.00 MB
  total, so W1 alone nearly closes it even without /GS-.
- **files:**
  1. `src/codegen/helpers.ts:34` (`declareASCIILiteral`) + `src/codegen/bundle-modules.ts:384,401`
     — emit `static constexpr const unsigned char ${name}Zstd[K] = {..};` +
     `static constexpr unsigned ${name}RawSize = N;` instead of the raw array. The
     generator already runs under bun so the compression is one `require("node:zlib")`.
  2. `src/jsc/bindings/InternalModuleRegistry.cpp:118` (the release
     `INTERNAL_MODULE_REGISTRY_GENERATE` macro) + `:36` `generateModule` — a ~20-line
     helper: `size_t n = ZSTD_getFrameContentSize(z, k); std::span<LChar> out;
     auto impl = StringImpl::createUninitialized(n, out); ZSTD_decompress(out.data(),
     n, z, k); RELEASE_ASSERT(n == rawSize);` then `makeSource(String(WTFMove(impl)),
     ...)` exactly as today. One `extern "C"` declaration of `ZSTD_decompress` /
     `ZSTD_getFrameContentSize` (already linked — F5).
  The debug `#ifdef BUN_DYNAMIC_JS_LOAD_PATH` from-disk branch at `:114` is untouched.
  No oven-sh/WebKit change.
- **effort:** small-medium (one codegen TS change + ~25 lines of C++; zero new deps).
- **relink_only:** NO — full rebuild (a generated header + one .cpp change). But
  verifiable on a DEBUG build too (`bun bd test test/js/node/`): the 158 modules must
  all still load.

  three -> 11,033 B total. 56,816 - 11,033 = **45,783 B = 0.0437 MB**.
- **confidence:** HIGH. The ONLY consumer is the `bun completions` / `bun getcompletes`
  CLI command.
- **risk:** NONE. A terminal, cold, interactive CLI path.
- **perf:** neutral (the only callers are `bun completions`/`bun getcompletes`, which
  then write the bytes to disk/stdout — 12 us of decompress on a one-shot I/O-bound
  command is unmeasurable).
- **regression:** none.
- **windows:** YES, ~0.044.
- **files:** `src/runtime/cli/shell_completions.rs:10-12` (replace the three
  `include_bytes!` with zstd blobs generated by a tiny build step, or `include_bytes!`
  of pre-compressed `.zst` files checked in next to the originals) + a ~10-line
  decompress at the two read sites.
- **effort:** small.
- **relink_only:** NO (one .rs).

#### W4 — `bun-sections-nobits`: make the all-zero `.bun` and `.bun_err` sections NOBITS

- **saving_mb:** **0.018** (+ up to ~0.011 of now-unneeded intra-RW-segment alignment
  padding; I conservatively claim only the 0.018). Derivation: `objcopy -O binary
  --only-section` on the real canary shows `.bun` = 16,384 B with 0 nonzero bytes and
  `.bun_err` = 2,788 B with 0 nonzero bytes. 19,172 B = **0.0183 MB**.
- **confidence:** HIGH on the bytes; MEDIUM on the `.bun` half's safety (see risk).
- **risk:** MEDIUM for `.bun`: `src/exe_format/elf.rs:469` and
  `src/jsc/bindings/c-bindings.cpp:1112` look the `.bun` section up BY NAME and the
  `bun build --compile` writer grows/patches it. If it becomes `SHT_NOBITS`, the writer
  must convert it back to PROGBITS when injecting the module graph — a small `elf.rs`
  change that MUST be tested with `bun build --compile` before/after (this is EXACTLY
  the failure mode that killed UPX, so the `--compile` round-trip test is mandatory).
  `.bun_err` (2,788 B) has no such consumer and is trivially safe.
  LOW-RISK ALTERNATIVE for most of the bytes with ZERO `--compile` risk: change
  `BLOB_HEADER_ALIGNMENT` at `c-bindings.cpp:1059` from `16*1024` to `8` (the comment
  says only `data[]`'s 8-byte alignment matters), shrinking the placeholder from
  16,384 B to ~16 B — IF the 16 KB alignment is not load-bearing for the writer.
  Git-blame it first.
- **perf:** neutral (NOBITS pages are zero-filled by the kernel exactly as mapped
  zero PROGBITS pages are; the runtime reads `BUN_COMPILED.size == 0` either way).
- **regression:** none, gated on the `--compile` round-trip test.
- **windows:** `.bun_err` is ELF-only (`cfg(target_os = "linux")`). `.bun` on PE is a
  separate path (`c-bindings.cpp:1091+`); not claimed.
- **files:** `src/jsc/bindings/c-bindings.cpp:1059,1078`; `src/bun_core/lib.rs:1135-1146`;
  possibly `src/exe_format/elf.rs:469`.
- **effort:** small.
- **relink_only:** close — a one-attribute change + relink; minutes to measure.

#### W5 (A-GATED) — `jsbuiltins-combined-zstd`: zstd the two combined JS-builtin source buffers

- **saving_mb:** **0.321** combined (0.207 bun-side + 0.113 WebKit-side). Derivation
  BY ADDRESS: `WebCore::combinedSourceCodeBuffer` 252,941 B @ 21,629,312 -> zstd-19
  35,478; `JSC::s_JSCCombinedCode` 136,133 B -> zstd-19 17,503. 389,074 - 52,981 =
  336,093 B = **0.3205 MB**. SUPERSEDES Tier-A row 18 (EJ2, 0.038) and Tier-B row B7
  (0.025); net-new over the inventory = 0.321 - 0.063 = **+0.26 MB**.
- **confidence:** MEDIUM on the perf gate; HIGH on the bytes.
- **WHY THIS IS GATED AND NOT TIER-A (the honest reason W1 is different):** Unlike the
  158 modules, JSC's `BuiltinExecutables` constructs its `SourceCode` over the combined
  buffer at **VM init** and several of its `@`-builtins ARE eagerly linked into the
  global object's structures. A whole-blob decompress at `BuiltinExecutables`
  construction costs ~250 us (derived from F4's measured 1.46 ms/MB) on EVERY bun
  invocation. That is a startup regression unless (a) one startup benchmark shows it is
  within noise, or (b) the decompress is overlapped on a background thread launched at
  the top of `main()` (works, but is a design a reviewer may reject). I could not prove
  neutrality by construction, so per the hard constraint this is A-GATED, not Tier-A.
  **Cost to find out: one rebuild + `hyperfine 'bun -e null'`.**
- **perf:** one-time (per VM), but at VM INIT, not on a lazy branch -> GATED.
- **regression:** none if the gate passes.
- **windows:** yes, ~0.32.
- **files:** `src/codegen/bundle-functions.ts:453-454` (bun) + oven-sh/WebKit
  `Source/JavaScriptCore/Scripts/wkbuiltins/builtins_generate_combined_implementation.py`
  (`s_JSCCombinedCode`) + `BuiltinExecutables.cpp`'s constructor.
- **effort:** medium (two repos).
- **relink_only:** NO.

#### W6 (LOW confidence) — `yarr-kstrings-lazy`: the 7 `JSC::Yarr::kStrings*Data` Unicode-property tables

- **saving_mb:** up to **0.085** (89,568 B across 7 `Yarr::kStrings*Data` tables, by
  address; the two >= 24 KB zstd to 3,867 + 2,302 B). This family is generated by
  oven-sh/WebKit's `GenerateYarrUnicodePropertyTables.py` and is DISJOINT from Tier-B
  row B4's two 64 KB bool tables (`_wordcharData`/`_spacesData`).
- **confidence:** LOW. I did NOT prove these are read only at regex COMPILE time (a
  genuinely one-time-lazy path per pattern). If Yarr reads them at MATCH time,
  lazy-decompress is a per-match branch = perf regression, and a .bss+startup-init is
  a startup regression. This needs ONE disassembly pass by whoever owns Yarr. Offered
  as a NAMED, SIZED lead, not a banked number.
- **perf:** UNKNOWN until the read site is classified -> gated.
- **windows:** yes.
- **files:** oven-sh/WebKit `Source/JavaScriptCore/yarr/generateYarrUnicodePropertyTables.py`.
- **effort:** medium.
- **relink_only:** NO.

#### W7 (Tier C — a product decision, listed because the brief gives explicit license)

- **`react-compiler-out-of-band`: 1.05 MB.** `bun_react_compiler`'s .text, re-derived
  BY ADDRESS from nm-dem.txt = **1.0532 MB across 1,139 unique addresses** (the census's
  0.88 was by name). It backs exactly ONE opt-in, documented flag
  (`bun build --react-compiler`). Mechanism: ship it as a separately-downloaded
  component (the way bun already fetches platform packages), dlopen'd on first use; or
  a `cfg` that a future `bun-full` variant enables. ZERO perf cost to the 99.x% of
  invocations that never pass the flag. **Why it will probably be rejected:** it is a
  second file, against bun's single-binary value proposition, and Jarred already
  rejected the adjacent `bun-standalone` split. The maintainers ALREADY care (it has a
  per-package `opt-level="s"` in Cargo.toml) and chose to keep it. Logged honestly as
  the single largest Tier-C candidate that nobody has formally filed; I do not
  recommend it. effort: large. windows: yes.
